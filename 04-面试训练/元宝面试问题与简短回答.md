# 元宝面试问题与简短回答

## 使用说明

- 来源对话: 与项目最贴合的面试集训
- 适用方向: 元宝客户端测试 / AI 应用测试面试。
- 回答原则: 能从简历和睿思 AI Agent 项目流程支撑的, 给理想简短回答; 项目中找不到明确实现依据的, 标记为【思路】。
- 表达边界: 只站在测试人员角色, 不包装成算法研发或模型训练经历。

## 一、自我介绍与项目真实性

1. 请做一个 2 分钟自我介绍。
   - 我有 7 年多测试和测试开发经验, 覆盖接口、Web/APP、IoT、性能、自动化和 AI Agent 测试。最近主要做睿思 AI Agent 数字员工测试, 重点是从用户入口到鉴权、RAG、Prompt/State、工具调用、最终回答的全链路质量保障。

2. 你为什么从传统测试转到 AI Agent 测试?
   - 因为 AI 产品质量不只看功能通不通, 还要验证事实正确性、权限边界、证据链、工具调用和幻觉风险, 和我之前做接口、日志、链路定位的经验能衔接。

3. 你在睿思 AI Agent 项目中具体负责哪些部分?
   - 我主要负责需求评审、测试方案、功能/集成测试、AI 效果评测、上线验收和回归, 重点测权限、意图分流、RAG、工具调用、异常返回和 TraceID 定位。

4. 这个项目里哪些是你亲自测试的, 哪些只是了解?
   - 我亲自测的是链路验证、权限验证、RAG 命中、工具调用、异常返回和回归样本; 模型训练、算法调参不是我的职责, 我主要从应用质量角度验证。

5. 你不是算法背景, 怎么保障 AI 产品质量?
   - 我不做模型训练, 但能把 AI 输出问题拆到接入、权限、RAG、Prompt、工具、接口和客户端展示层, 用日志、样本和回归验证推动闭环。

6. 你觉得元宝客户端测试和你过往项目最大的关联是什么?
   - 元宝也是用户输入到 AI 回答的复杂链路, 我的项目经验可以迁移到客户端入口、会话、附件、RAG、工具调用、异常返回和线上问题定位。

7. 你简历里写 Web/APP/IoT/AI 测试, 哪段经历最能支撑元宝岗位?
   - 睿思 AI Agent 项目最贴近, 因为它既有客户端入口和附件场景, 又有 AI 链路里的 RAG、Prompt、工具调用、权限和结果生成。

8. 你做过最复杂的一次问题定位是什么?
   - 通过 TraceID 串接入口参数、权限、意图分流、RAG 召回、工具调用和最终回答, 判断问题具体发生在检索、工具、生成还是展示层。

9. 你有没有推动过研发修复 AI 输出质量问题? 怎么证明?
   - 可以说我通过 badcase 记录输入、TraceID、召回结果、Observation 和最终回答, 推动研发修复或回归验证。若没有具体数据, 不要说“提升 xx%”。

10. 你如何避免把 AI Agent 项目讲成“模型训练项目”?
    - 明确我的角色是测试, 重点讲链路、权限、RAG 证据、工具调用、异常处理、幻觉识别和回归样本, 不说自己训练模型。

## 二、AI Agent 主流程追问

1. 按你的流程图, 从用户输入到最终回答完整讲一遍。
   - 用户从 WeLink/Web/API 进入, UniKnow 接入和账号绑定, 请求保留消息、附件、TraceID, 经权限校验后进入 Race Agent Service, 再做意图识别、RAG、Prompt/State、工具调用和最终回答。

2. WeLink / Web / API 三个入口测试重点有什么不同?
   - WeLink 偏账号和会话, Web 偏页面交互和附件, API 偏参数、鉴权、错误码和自动化验证。

3. UniKnow 在链路里起什么作用?
   - 做统一接入、账号绑定和入口规范化, 保证后续权限和用户身份能识别。

4. 请求入口为什么要保留消息、附件、TraceID?
   - 为了复现用户输入、定位附件处理问题, 并串起后续 RAG、工具调用和最终回答日志。

5. 身份权限、工具权限、知识库权限分别怎么测?
   - 身份测登录有效/失效, 工具权限测能否调用工具, 知识库权限测不同部门是否只能召回有权限知识。

6. 鉴权失败后为什么不能继续进入 Agent 链路?
   - 否则可能造成越权知识召回、越权工具调用或敏感信息泄露。

7. Race Agent Service 主要负责编排什么?
   - 负责上下文、路由、任务分流和进入 Agent 后的执行编排。

8. 意图识别怎么区分普通问答、固定流程、复杂 Agent 任务?
   - 普通问答不进工具链路, 固定流程走标准流程, 复杂任务如日志分析、带附件处理才进入 Agent 规划和工具调用。

9. 普通问答误进 Agent 会带来什么风险?
   - 增加耗时、误调用工具、引入不必要的 RAG 或错误结果。

10. 复杂任务没有进入 Agent 会出现什么问题?
    - 只能给泛化回答, 不能结合附件、工具和业务状态做分析。

11. 上下文是否足够怎么判断?
    - 看目标、参数、附件、权限是否满足执行条件。

12. 缺参数、缺附件、缺权限分别怎么构造用例?
    - 如“帮我分析一下”但没附件, 上传日志但没说明目标, 工具需要业务域但输入缺失。

13. 单轮 Agent 缺参结束和多轮对话有什么区别?
    - 单轮缺参后只返回澄清并结束, 不等待用户补充后自动恢复。

14. 用户补充参数后, 为什么要作为新请求重新进流程?
    - 因为流程约束是下一次输入作为新请求重新鉴权、判断上下文和路由。

15. 如果系统继续沿用上一轮未完成状态, 可能有什么缺陷?
    - 可能串会话、错用旧附件、绕过权限或错误调用工具。

## 三、RAG 测试追问

1. 什么问题需要走 RAG?
   - 企业知识、规范、流程、接口文档、产品说明、内部字段、工具参数类问题。

2. 什么问题不应该走 RAG?
   - 实时接口查询、纯工具执行、当前输入已足够的问题。

3. RAG 误触发会有什么问题?
   - 引入无关知识, 导致回答跑偏或变慢。

4. RAG 漏触发会有什么问题?
   - 回答缺少依据, 容易幻觉。

5. Query Rewrite 怎么测?
   - 对比原问题、改写 query、召回结果, 看是否保留原意和业务域。

6. 改写后的 query 偏离用户原意怎么发现?
   - 看改写后是否扩大范围、丢失字段名、丢失附件上下文或召回无关片段。

7. 权限过滤应该发生在哪个阶段?
   - 在召回前或召回过程中生效, 至少必须在知识进入 Prompt 前生效。

8. 为什么无权限知识不能进入 Prompt?
   - 模型一旦看到无权限内容, 最终回答就有泄露风险。

9. TopK 召回怎么评估?
   - 看相关候选是否被召回, 尤其是标准答案片段是否进入候选集。

10. ReRank 怎么评估?
    - 看最有用证据是否排到前面, 无关或冲突证据是否被降权。

11. TopK 命中了但最终回答错, 问题可能在哪里?
    - 可能是 ReRank、Prompt 注入、生成总结或最终回答阶段的问题。

12. TopK 没命中但模型答出来了, 你怎么判断是否幻觉?
    - 如果没有其他可靠来源, 要按幻觉风险处理。

13. RAG 未命中时最终回答应该怎么表现?
    - 应说明知识不足或无法确认, 不能编造内部制度、流程或字段。

14. 知识库里有冲突知识时怎么测?
    - 构造新旧版本、同名字段、不同部门知识, 看是否选择正确证据并说明依据。

15. 知识过期、字段改名、流程变更怎么回归?
    - 回归 Query Rewrite、权限过滤、TopK、ReRank、未命中和最终回答一致性。

## 四、Prompt / State / Agent 执行追问

1. Prompt / State 里应该包含哪些内容?
   - 用户输入、附件解析、RAG 证据、权限状态、上下文和工具结果。

2. State 缺少附件解析结果会怎样?
   - 日志分析会变成泛化回答, 或错误判断没有附件。

3. State 缺少权限状态会怎样?
   - 可能越权回答或越权调用工具。

4. State 缺少 RAG 证据会怎样?
   - 企业知识类问题容易无依据回答。

5. Agent Executor 的 Plan / Thought / Action / Observation 怎么理解?
   - Plan 是计划, Action 是执行动作, Observation 是工具或环境返回结果; 测试重点是链路是否合理。

6. 测试人员是否需要看 Thought?
   - 不评价模型思考本身, 重点看可观测的计划、动作、工具调用、Observation 和结果是否一致。

7. Action 选择错误怎么定位?
   - 看意图识别、Tool Selection、权限、Schema 和上下文参数。

8. Observation 记录错误但最终回答成功, 这是什么问题?
   - 说明结果判断或最终生成有问题, 可能编造成功结果。

9. 工具结果不足时系统应该继续规划还是终止?
   - 如果缺关键证据或工具失败, 应返回失败原因或缺失信息, 不应强行下结论。

10. Prompt 变更后为什么要回归工具选择和缺参澄清?
    - Prompt 会影响是否澄清、是否调用工具、是否拒答和最终表达。

11. Prompt 变更后如何设计回归集?
    - 用历史缺陷、高频问题、权限边界、RAG 未命中、工具失败、缺参样本。

12. 如何判断 AI 回答是“事实正确”而不是“表达流畅”?
    - 对照 RAG 证据、工具结果、业务规则和关键事实点。

13. 非确定性回答怎么断言?
    - 不做全文匹配, 按事实点、证据、权限、错误类型和风险提示断言。

14. 如何做 AI badcase 沉淀?
    - 记录输入、TraceID、权限、RAG、工具结果、Observation、最终回答和期望结果。

15. 如何评估一次 Prompt 改动有没有引入回归?
    - 跑固定样本集, 对比缺参、拒答、工具选择、幻觉和最终回答质量。

## 五、工具调用测试追问

1. Tool Selection 怎么测?
   - 构造不同意图、权限和参数样本, 看是否选择正确工具。

2. 纯知识问答误选工具是什么问题?
   - 属于意图分流或工具选择错误, 会增加风险和耗时。

3. 有工具权限和无工具权限怎么覆盖?
   - 有权限用户能调用, 无权限用户被拦截, 且不进入 Tool Call。

4. 工具 Schema 变更后要回归哪些点?
   - 回归参数名、必填项、类型、枚举、默认值、错误返回。

5. 参数名、类型、枚举变化会影响什么?
   - 可能导致调用失败、入参错、业务结果错或 Observation 错。

6. 参数来自用户输入、RAG、上下文时分别有什么风险?
   - 用户输入可能抽取错, RAG 可能引用错, 上下文可能串会话。

7. 工具必填参数缺失时应该怎么返回?
   - 应返回缺失参数或校验失败, 不应继续调用。

8. 工具超时、失败、空结果怎么测?
   - Observation 记录真实状态, 最终回答说明失败或结果不足。

9. API / MCP / 工作流 / 子 Agent 调用测试有什么差异?
   - API 关注入参出参, MCP 关注工具协议, 工作流关注节点状态, 子 Agent 关注上下游结果一致性。

10. 工具调用成功但业务结果错怎么定位?
    - 查入参、工具返回、业务规则和最终总结是否一致。

11. 工具失败后最终回答应该怎样?
    - 不能说已完成, 应说明失败原因、缺失信息或建议重试。

12. 如何判断 Agent 是否编造了工具执行结果?
    - 对比 Tool Call 日志、Observation 和最终回答。

13. 如何用 TraceID 串起 Tool Call 和最终回答?
    - 用同一 TraceID 查入口、Tool Call、Observation、最终回答日志。

14. 工具返回部分字段时, 最终回答能不能下结论?
    - 不能覆盖用户目标时不能下确定结论, 应说明结果不足。

15. 工具调用产生副作用时, 测试要注意什么?
    - 【思路】要验证权限、幂等、确认机制、执行结果和回滚/补偿, 避免误操作。

## 六、客户端测试追问

1. 元宝客户端的核心用户链路你会怎么测?
   - 登录、输入问题、上传附件、生成回答、查看历史、异常重试。

2. 登录、会话、提问、文件上传、图片上传、回答展示怎么拆用例?
   - 按正常、异常、权限、弱网、兼容、性能和日志定位拆。

3. 流式输出怎么测?
   - 测首 token、连续输出、中断、停止生成、重试和 UI 展示。

4. 回答中断、重试、停止生成怎么测?
   - 看请求是否取消、状态是否正确、重复点击是否安全。

5. 历史会话同步怎么测?
   - 测新增、删除、切换设备、弱网刷新和账号切换。

6. 多端登录、账号切换怎么测?
   - 测会话一致性、权限状态、消息同步和账号隔离。

7. 客户端版本升级后怎么回归?
   - 核心链路、兼容、缓存、历史数据、权限和崩溃。

8. 弱网下提问、上传附件、生成回答怎么测?
   - 测上传失败、生成超时、重试、断网恢复和错误提示。

9. 断网恢复后会话状态应该怎样?
   - 页面状态要可恢复, 不能丢失已提交内容或展示错误状态。

10. 前后台切换、锁屏、杀进程后怎么验证?
    - 测生成状态、上传状态、会话保存和重新进入展示。

11. 权限开关包括哪些? 相册、相机、麦克风、通知、文件访问怎么测?
    - 测相册、相机、麦克风、文件、通知等授权、拒绝、再次开启。

12. 不同机型、系统版本、屏幕尺寸怎么做兼容覆盖?
    - 按系统版本、主流机型、屏幕尺寸、网络类型和权限状态覆盖。

13. 客户端性能指标看哪些?
    - 启动、页面加载、首 token、卡顿、内存、耗电、发热。

14. 启动耗时、页面加载、卡顿、内存、耗电怎么定位?
    - 客户端看日志、耗时点、内存、帧率; 服务端看接口和生成耗时。

15. Crash / ANR 怎么收集和分析?
    - 收集堆栈、设备、系统版本、操作路径、日志和复现条件。

## 七、接口与日志定位追问

1. 用户反馈“回答不对”, 你第一步收集什么?
   - 问题描述、截图、时间、账号、入口、版本、会话 ID、TraceID。

2. 没有 TraceID 时怎么辅助定位?
   - 用账号、时间、入口、会话 ID、客户端日志和服务端日志交叉查。

3. 如何区分客户端问题、服务端问题、RAG 问题、模型问题?
   - 看请求参数、权限、RAG、工具、Observation、最终回答和页面展示。

4. 客户端请求参数错会有哪些表现?
   - 意图识别错、缺参、工具入参错、结果不符合用户目标。

5. 接口返回正常但页面展示异常怎么定位?
   - 偏客户端渲染、状态管理、缓存或兼容问题。

6. 接口超时和模型生成慢怎么区分?
   - 看服务端接口耗时、首 token 时间、流式输出时间。

7. 附件上传成功但 Agent 没读到附件怎么查?
   - 查附件 ID、解析状态、State 注入和 Agent 输入。

8. 权限接口通过但知识库召回为空怎么查?
   - 查知识库权限过滤、检索 query、TopK、索引和知识是否存在。

9. 错误码设计不清晰会影响测试什么?
   - 难以判断失败类型, 影响自动化断言和线上定位。

10. 埋点在 AI 客户端测试里有什么用?
    - 还原用户路径、定位操作步骤、分析失败率和转化漏斗。

11. 如何通过日志复现线上问题?
    - 根据输入、账号、权限、附件、TraceID 和环境重放。

12. 如何判断问题是偶现还是稳定复现?
    - 多次复现、不同账号/设备/网络对比, 看日志是否同一失败点。

13. 线上问题修复后怎么做回归?
    - 回归原 badcase、同类场景、权限边界和核心主链路。

14. 回归样本怎么沉淀?
    - 按问题类型归档, 加入回归集。

15. 如何给研发一个可复现的缺陷单?
    - 写清环境、账号、步骤、输入、实际/预期、TraceID、日志证据和影响范围。

## 八、AI 质量与幻觉追问

1. 什么是 AI 幻觉?
   - AI 在缺少依据、证据不足或工具失败时仍生成看似确定的错误内容。

2. 幻觉和普通功能 bug 有什么区别?
   - 普通 bug 多是确定逻辑错误, 幻觉常表现为流畅但无依据。

3. RAG 未命中但回答很肯定, 怎么判定?
   - 看 RAG 日志和最终回答, 无依据则按幻觉风险记录。

4. 工具失败但回答成功, 怎么判定?
   - 对比 Observation 和最终回答, 属于编造执行结果。

5. 权限不足但回答了内部知识, 属于什么风险?
   - 属于越权和信息泄露风险, 优先级高。

6. 回答部分正确、部分错误怎么评估?
   - 拆关键事实点, 标记错误事实和影响范围。

7. 多个证据冲突时最终回答怎么评估?
   - 看是否选择权威、最新、有权限证据, 是否说明不确定性。

8. 如何设计反幻觉测试集?
   - 缺参、无知识、工具失败、冲突证据、无权限、过期知识。

9. 如何测试拒答边界?
   - 测无权限、无依据、敏感信息、危险操作和超出范围问题。

10. 如何测试敏感信息泄露?
    - 用不同角色和部门账号验证知识、日志、附件是否隔离。

11. 如何测试越权访问?
    - 测无权限用户是否无法召回知识、选择工具或获得内部结果。

12. AI 回答每次不同, 自动化怎么做断言?
    - 用结构化断言, 不做全文匹配。

13. 事实类问题、总结类问题、日志分析类问题断言有什么不同?
    - 事实类看准确性, 总结类看要点覆盖, 日志类看错误点和建议。

14. 历史 badcase 怎么用于回归?
    - 每次 Prompt/RAG/工具变更后跑历史 badcase。

15. 你如何证明 AI 质量有提升?
    - 【思路】用固定评测集对比通过率、幻觉率、拒答正确率、工具调用准确率, 但不要编造简历里没有的数据。

## 九、自动化与性能追问

1. 你用 Python / Pytest 做过哪些自动化?
   - 主要用于接口自动化、回归测试、参数化用例和断言接口返回。

2. AI Agent 流程哪些适合自动化, 哪些不适合?
   - 权限、接口、RAG 命中、工具调用、错误码、历史 badcase 适合; 主观开放回答需人工或半自动评估。

3. 接口自动化如何覆盖权限、RAG、工具调用?
   - 构造不同账号、权限、query、附件和工具参数, 校验日志节点和返回。

4. 非确定性回答如何做自动化断言?
   - 断关键事实点、状态、证据、工具调用、错误类型, 不断完整文案。

5. Prompt 回归集怎么组织?
   - 按正常、缺参、权限、RAG 未命中、工具失败、历史缺陷分类。

6. Locust 做过什么性能测试?
   - 可说明用于接口性能和稳定性压测, 关注并发、响应时间、失败率。

7. AI 问答接口性能关注哪些指标?
   - 首 token 时间、总生成时间、接口耗时、失败率、超时率。

8. 首 token 时间、总生成时间、接口超时怎么测?
   - 客户端和服务端分别打点, 区分排队、检索、工具调用、生成耗时。

9. 并发用户提问时如何验证稳定性?
   - 看错误率、超时、资源占用、RAG/工具瓶颈和回答完整性。

10. 附件上传 + Agent 分析的性能瓶颈可能在哪里?
    - 可能在上传、解析、RAG、工具调用或模型生成。

11. RAG 检索慢怎么定位?
    - 查 Query Rewrite、检索、权限过滤、ReRank 各阶段耗时。

12. 工具调用慢怎么定位?
    - 查 Tool Call 入参、接口耗时、超时、下游服务状态。

13. 客户端卡顿和服务端响应慢怎么区分?
    - 客户端看帧率、主线程、内存; 服务端看接口和生成耗时。

14. Jenkins 如何接入回归?
    - 用 Jenkins 定时或提测触发自动化回归, 输出报告和失败用例。

15. 自动化失败后如何判断是脚本问题还是产品问题?
    - 先看环境和脚本稳定性, 再查产品日志、接口返回和最近变更。
