面试官问题:

请你做一个 1 分钟左右的自我介绍。重点说清楚:

1. 你的测试经历主线
2. 你最近做的 AI Agent 项目
3. 为什么这个经历和元宝客户端测试岗位匹配

可以这样答:

面试官你好, 我叫宋克宇, 2018 年毕业于内蒙古科技大学计算机科学与技术专业, 毕业后一直从事软件测试工作, 到现在有七年多经验。

2019 年到 2022 年, 我在天猫精灵相关项目中主要负责功能测试、接口测试, 也做过一些自动化工具和脚本开发。2022 年之后, 我通过 OD 项目参与华为内部业务系统测试, 主要负责 Web 平台的需求评审、测试方案设计、功能测试、接口测试、缺陷跟踪和版本回归。

最近团队在做一个 AI Agent 数字员工项目, 它是基于公司内部 Agent 平台搭建的, 会把原来的一些业务工具封装成 Tool 接入 Agent, 用户通过自然语言发起任务, Agent 再做意图识别、参数提取、工具选择和结果返回。

我在项目里主要从测试角度关注整条链路的质量, 包括接入鉴权、权限校验、RAG 召回、工具调用、异常返回、TraceID 日志定位以及最终回答是否有依据。这个经历和元宝客户端测试比较匹配, 因为元宝也不只是测客户端页面, 还要关注用户输入、会话、接口链路、AI 回答质量和线上问题定位。

### 问题 2: 用户从 WeLink 发起问题后, 完整链路是怎样的

面试官问题:

你刚才说这个 Agent 是“数字员工”。请你具体讲一下: 用户从 WeLink 发起一个问题后, 这个请求到最终返回答案, 中间完整链路是怎样的?

可以这样答:

用户从 WeLink 发消息后, 请求先进入 UniKnow 平台, UniKnow 主要负责统一接入和消息转发, 在这里可以看到用户输入、最终回复以及相关 TraceID。

请求转到我们自己的服务后, 会先做鉴权, 包括用户身份、工具权限和知识库权限。如果无权限, 会直接返回无权限提示。鉴权通过后进入意图分流, 大致分为通用问答、知识库问答和 Agent 工具调用。

如果是通用问答, 可以由模型直接回复。如果是知识库问答, 会进入 RAG 链路, 做权限过滤、召回、重排, 再把可用知识注入上下文生成回答。如果是工具调用类任务, 会先判断用户目标、参数、附件和权限是否足够, 然后由 Agent 做工具选择、参数组装、工具调用, 工具结果进入 Observation, 最后模型基于工具结果总结后返回给用户。

### 问题 3: 线上用户反馈 Agent 回答错了, 怎么定位

面试官问题:

如果线上用户反馈“Agent 回答错了”, 你拿到用户截图和反馈时间后, 你会怎么一步步定位?重点说清楚 TraceID 怎么用, 以及你怎么区分是客户端问题、RAG 问题、工具问题, 还是模型总结问题。

可以这样答:

我会先收集用户反馈时间、账号、截图、客户端版本、会话信息和问题文本。然后去 UniKnow 按时间、账号或问题文本模糊匹配日志, 找到对应请求的 TraceID。

拿到 TraceID 后, 我会串入口日志、鉴权日志、意图分流日志、RAG 日志、工具调用日志、Observation 和最终回复。

如果 UniKnow 没有收到消息, 我会优先看客户端是否发起请求、网络状态、附件上传是否成功、接口返回码和客户端日志, 判断是不是客户端或接入问题。

如果消息进入了服务, 但意图分流错误, 比如工具调用问题被分到普通问答, 就是意图识别问题。

如果是 RAG 类问题, 我会看知识库里是否有正确知识, query rewrite 有没有改偏, TopK 是否召回正确片段, ReRank 是否把有效片段排到前面, 最终 Prompt 里是否注入了正确证据。知识库有但没召回, 偏 RAG 召回问题; 召回了但模型答错, 偏模型总结问题。

如果是工具类问题, 我会看工具选择是否正确, 参数是否提取正确, Tool Call 是否成功, Observation 是否记录了正确结果。如果工具返回就是错的, 是工具或下游接口问题; 如果工具返回正确但最终回答错, 就是模型总结或 Prompt 问题。

如果 RAG 未命中、工具失败, 但最终回答还给了确定结论, 我会把它归为幻觉风险 badcase。

### 问题 4: 知识库里明明有答案, 但 Agent 最终回答错了, 怎么定位 RAG 问题

面试官问题:

如果是 RAG 类问题: 用户问的问题知识库里明明有, 但 Agent 最终回答错了。你会怎么区分是 Query Rewrite 问题、TopK 召回问题、ReRank 问题、还是模型生成问题?

可以这样答:

我会按 RAG 链路逐段看证据。

第一看 Query Rewrite, 对比用户原始问题和改写后的 query。如果改写后丢了关键字段、业务域或把用户问题改偏, 那就是 Query Rewrite 问题。

第二看 TopK 召回。如果知识库里有正确内容, 但 TopK 候选里完全没有相关片段, 就是召回问题。

第三看 ReRank。如果 TopK 里已经有正确片段, 但 ReRank 后正确片段没有进入最终上下文, 或被噪声片段挤掉, 就是重排问题。

第四看最终注入给模型的上下文。如果 Prompt 里已经包含正确证据, 但最终回答仍然答错或编造, 那更偏模型生成或 Prompt 约束问题。

定位时我会把原始问题、rewrite query、TopK 结果、ReRank 结果、注入片段和最终回答都记录下来, 作为 RAG badcase 进入回归集。

### 问题 5: 怎么验证知识库权限隔离

面试官问题:

如果两个用户问同一个知识库问题, A 有权限, B 没权限, 你会怎么设计测试用例来验证知识库权限隔离?输出你会覆盖哪些场景, 以及你期望看到什么结果。

可以这样答:

我会设计一组同题不同账号的权限隔离用例。

第一类是正向用例: A 用户有知识库权限, 提问目标知识相关问题, 期望 RAG 能召回正确片段, 最终回答基于有权限的知识返回。

第二类是反向用例: B 用户没有该知识库权限, 提同一个问题, 期望在权限过滤后召回不到该知识, Prompt/State 里不能出现该知识内容, 最终回答不能泄露答案, 可以提示无权限或无法确认。

第三类是角色和部门隔离: 使用不同部门、不同角色账号问同一个问题, 验证只能召回自己权限范围内的知识。

第四类是绕过类问题: B 用文档别名、关键词、同义问法、摘要式提问来问, 也不能召回或泄露无权限内容。

验证时我不会只看最终回复, 还会用 TraceID 看权限校验结果、RAG 召回结果、最终注入上下文和最终回答, 确认无权限知识没有进入模型上下文。

### 问题 6: 无权限用户没有看到原文, 但模型总结出了无权限知识结论, 算不算权限问题

面试官问题:

如果无权限用户 B 最终没有直接返回原文, 但模型根据无权限知识总结出了结论, 比如“这个流程需要主管审批”, 你认为这算不算权限问题?为什么?

可以这样答:

我认为算权限问题。因为权限隔离不只是不能返回原文, 也不能基于无权限知识做总结、改写或推理。

如果 B 没有权限, 正确结果应该是在召回阶段就过滤掉这部分知识, Prompt/State 中也不能出现相关内容, 最终只能提示无权限或无法确认。

如果模型回答了“这个流程需要主管审批”, 即使没有暴露原文, 也说明无权限知识可能已经进入了上下文, 属于知识泄露风险。定位时我会用 TraceID 看权限过滤结果、RAG 召回片段和最终注入 Prompt 的内容, 确认是哪一段没有拦住。

### 问题 7: 没有上传附件却调用日志分析工具, 怎么判断和回归

面试官问题:

如果用户说“帮我分析这份日志”, 但是没有上传附件, Agent 却仍然调用了日志分析工具并返回了一个分析结论。你会怎么判断这个问题?你会设计哪些用例防止这类问题再次出现?

可以这样答:

这个问题我会先判定为异常, 因为“分析日志”依赖附件, 没有附件时 Agent 不应该继续调用工具, 更不能返回确定分析结论。

我会用 TraceID 查看前置判断、参数提取、Tool Call 入参、工具返回和最终回答。

如果 Tool Call 入参里没有附件 ID, 但 Agent 仍然发起调用, 说明 Agent 的缺参判断或参数校验有问题。

如果工具在附件为空时仍然返回了分析结论, 说明工具侧必填参数校验也有问题。

如果入参里出现了历史会话里的附件 ID, 那要重点查上下文串会话或状态污染。

回归用例上, 我会覆盖: 未上传附件直接要求分析, 上传附件但不说明目标, 上传错误格式附件, 多轮会话中上一轮有附件但本轮没有附件, 以及附件 ID 为空时工具是否拒绝执行。期望结果是缺附件时只提示用户补充附件, 不进入工具调用, 更不能生成分析结论。

### 问题 8: 工具失败但最终回答成功, 怎么定性和回归

面试官问题:

如果工具接口实际返回失败, Observation 里也记录了失败, 但最终回答却说“分析完成, 未发现异常”。你会怎么定性这个问题?后续怎么回归?

可以这样答:

我会把它定性为模型总结问题和幻觉风险。因为工具接口返回失败, Observation 也记录失败, 说明执行链路已经知道任务没有成功, 但最终回答却给了“分析完成、未发现异常”的确定结论, 这是最终回复没有基于 Observation 生成。

后续回归我会重点覆盖工具失败类样本, 比如接口超时、错误码、空结果、权限失败、参数校验失败、下游服务异常。期望最终回答必须说明失败原因或建议重试, 不能生成成功结论。

同时我会检查 Tool Call、Observation、最终回答三者是否一致, 把这类失败样本加入 Prompt 变更和工具变更后的回归集。

### 问题 9: Prompt 改版后回归哪些重点场景

面试官问题:

如果 Prompt 做了一次改版, 产品说“只是优化表达方式, 不影响逻辑”。作为测试, 你会回归哪些重点场景?为什么不能只测正常问答?

可以这样答:

我不会只按“表达优化”处理, 因为 Prompt 改动可能影响模型的意图识别、工具选择、参数提取、RAG 使用、拒答策略和最终回答边界。

回归时第一类是核心正常链路, 比如通用问答、知识库问答、工具调用任务, 确认主流程没有退化。

第二类是异常和边界场景, 包括缺参数、缺附件、无权限、RAG 未命中、工具失败、接口超时、空结果、冲突证据。

第三类是历史 badcase 和高频线上问题, 尤其是之前出现过幻觉、误选工具、越权回答、失败却回答成功的样本。

第四类是针对 Prompt 改动点做单点验证, 看新表达是否真的生效, 有没有影响原有约束。

不能只测正常问答, 因为正常链路往往最容易通过, 真正能暴露 Prompt 风险的是拒答、澄清、权限、失败处理和无依据回答这些边界场景。

### 问题 10: 元宝这类 AI 客户端怎么设计测试点

面试官问题:

现在切到客户端测试。元宝这类 AI 客户端, 如果让你设计测试点, 你会从哪些维度覆盖?请按功能、兼容、性能、稳定性、弱网/异常、日志定位这几个方向回答。

可以这样答:

功能方面, 我会覆盖登录、会话创建、文本输入、图片或文件上传、历史记录、流式回答、中断重试、复制分享、清空会话、异常提示等核心功能, 同时验证客户端展示和服务端返回是否一致。

兼容方面, 我会覆盖不同系统版本、机型、分辨率、浏览器或客户端版本, 重点看页面布局、输入框、附件上传、权限弹窗、流式渲染和历史记录展示是否正常。

性能方面, 我会关注启动耗时、页面加载耗时、接口响应、首包时间、流式输出速度、图片或文件上传耗时、内存占用、卡顿、耗电等指标。

稳定性方面, 我会验证长时间对话、多轮连续提问、大文件上传、频繁切后台、锁屏恢复、网络切换、低电量等情况下是否出现崩溃、ANR、卡死、会话丢失或重复发送。

弱网和异常方面, 我会构造断网、弱网、网络抖动、接口超时、服务端 5xx、附件上传失败、流式回答中断等场景, 看客户端是否有明确提示, 是否支持重试, 是否出现页面错乱或状态不一致。

日志定位方面, 我会关注客户端日志、接口状态码、请求参数、会话 ID、TraceID、客户端版本、设备信息和埋点。线上问题定位时, 需要能从客户端请求串到服务端链路, 判断问题发生在客户端展示、接入接口、RAG、工具调用还是模型返回。

### 问题 11: RAG 未命中时应该怎么处理

面试官问题:

如果用户问了一个依赖企业知识库的问题, 但 RAG 没有召回到有效内容, 你认为系统应该怎么处理?

追问:

1. 怎么防止模型编造答案?
2. 你会设计哪些用例验证 RAG 未命中逻辑?
3. 如果最终回答很流畅, 但实际上没有任何知识依据, 你怎么定性?

可以这样答:

如果用户问题依赖企业知识库, 但 RAG 没有召回到有效内容, 系统不应该编造制度、流程或接口说明, 应该提示“当前知识库未找到依据, 无法确认”或者引导用户补充资料。

如果这个问题本身不依赖企业知识, 可以按普通问答处理, 但不能伪装成基于知识库回答。

防止幻觉不能只靠模型自觉, 需要把 RAG 未命中的状态传到 Prompt/State 里, 并在 Prompt 中约束没有依据时不能给确定结论。同时测试要检查最终回答是否真的遵守这个状态。

用例上我会覆盖: 知识库不存在的问题、过期术语、无权限知识、相似但不正确的知识、RAG 召回为空、TopK 有结果但相关性低的情况。验证时看 RAG 日志、TopK 结果、最终注入上下文和最终回答。如果没有有效证据但回答给了确定结论, 就定性为模型幻觉风险, 需要进入 badcase 回归集。

### 问题 12: Prompt / State 构建怎么测

面试官问题:

Agent 执行前会把用户输入、附件、RAG 证据、权限状态等内容构造成 Prompt / State。你会怎么测试 Prompt / State 构建是否正确?

追问:

1. State 缺少 RAG 证据会有什么后果?
2. State 缺少权限信息会有什么后果?
3. State 缺少附件解析结果会有什么后果?

可以这样答:

我会把 Prompt / State 当成 Agent 执行前的上下文输入来验证, 重点看用户原始输入、附件解析结果、RAG 证据、权限状态、历史上下文和工具约束是否完整、准确地进入 State。

验证时我会通过 TraceID 或调试日志查看 State 内容, 对比用户输入、RAG 召回片段、权限校验结果和附件解析结果, 确认没有缺失、串会话或注入无权限内容。

如果 State 缺少 RAG 证据, 模型可能只能泛化回答, 甚至编造企业制度或接口说明。如果 State 缺少权限信息, 可能导致越权回答或越权工具调用。如果 State 缺少附件解析结果, 日志分析、图片识别或文件问答就可能变成空泛回答。

所以测试用例会覆盖有 RAG、有附件、有权限限制、缺附件、无权限、历史会话切换等场景, 看 State 和最终回答是否一致。

### 问题 13: Agent Executor 的 Plan / Thought / Action / Observation 怎么测

面试官问题:

Agent 进入执行阶段后, 通常会有 Plan / Thought / Action / Observation 这类中间过程。你作为测试人员, 会怎么验证这条执行链路是合理的?

追问:

1. Plan 主要看什么?
2. Action 主要看什么?
3. Observation 主要看什么?
4. 如果工具返回失败, 但后续仍然生成成功结论, 你怎么定位?

可以这样答:

我测试 Plan / Action / Observation 时, 重点不是评价模型思考写得好不好, 而是验证执行链路是否符合用户目标和业务规则。

Plan 主要看任务拆解是否覆盖用户目标, 有没有遗漏关键步骤, 有没有把不该做的事情规划进去。

Action 主要看选择的工具是否正确, 入参是否完整, 是否符合权限和工具 Schema, 有没有越权调用或误选工具。

Observation 主要看工具调用后的结果是否被正确记录, 包括成功结果、失败状态、错误码、空结果和业务异常。

后续生成回答时, 应该基于 Observation 决策。如果工具返回失败, Observation 也记录失败, 但最终回答仍然说成功, 我会定性为最终总结没有遵守 Observation, 属于模型幻觉或结果判断问题。定位时会用 TraceID 串 Plan、Action、Tool Call、Observation 和最终回答。

### 问题 14: Tool Selection 怎么测

面试官问题:

Agent 有多个工具可选时, 你会怎么验证它没有选错工具?

追问:

1. 如何判断误选工具?
2. 如何判断越权调用?
3. 工具名称或能力相似时, 你怎么设计用例?

可以这样答:

我会通过 Agent 的执行日志或 ReAct 过程查看 Tool Selection 结果, 再用 TraceID 串到实际 Tool Call 日志, 确认调用的工具 API 是否和用户意图一致。

判断误选工具时, 不只看最终回答, 还要看用户意图、工具描述、工具 Schema、入参和实际调用链路。比如纯知识问答不应该调用日志分析工具, 日志分析任务应该选择日志分析工具, 数据查询任务不能误选处理类工具。

判断越权调用时, 我会用无权限账号构造工具调用任务, 期望权限校验阶段或 Tool Selection 阶段拦住。如果用户没有 A 工具权限, 但最终 Tool Call 成功调用了 A 工具, 就是越权调用。

对于工具名称或能力相似的情况, 我会设计最小差异样本和交叉样本。比如用 A 工具的业务描述搭配 B 工具参数, 或用相似关键词但不同业务域的问题, 看 Agent 是否依据真实意图和参数选择正确工具。验证时看 Tool Selection 日志、Tool Call API、入参和最终 Observation 是否一致。

### 问题 15: 参数组装与校验怎么测

面试官问题:

Agent 选中工具后, 需要从用户输入、RAG 结果、上下文中提取参数并组装 Tool Call 入参。你会怎么测参数组装与校验?

追问:

1. 参数来自用户输入时要注意什么?
2. 参数来自 RAG 时要注意什么?
3. 参数来自上下文时要注意什么?
4. 如果必填参数缺失, 系统应该怎么处理?

可以这样答:

我会重点验证参数来源、必填项、类型、枚举、业务域和权限是否正确。

参数来自用户输入时, 要看模型是否正确抽取关键字段, 比如设备类型、日志文件 ID、时间范围、业务域, 不能把用户的口语化描述抽错或漏抽。

参数来自 RAG 时, 要看引用的知识是否和当前问题相关, 不能把相似但无关的知识片段当成工具参数, 也不能引用无权限知识。

参数来自上下文时, 要重点防止串会话或状态污染, 比如上一轮的附件 ID、业务域、账号信息不能错误带到本轮任务。

如果必填参数缺失, 系统不应该继续调用工具, 应该返回缺失信息或参数校验失败原因。验证时我会看 Tool Call 入参、参数校验日志、Observation 和最终回答是否一致。

### 问题 16: Tool Call 和 Observation 怎么测

面试官问题:

Tool Call 和 Observation 怎么测? 工具失败后最终回答应该怎样?

可以这样答:

Tool Call 主要验证是否调用了正确的 API、MCP、工作流或子 Agent, 入参是否完整, 权限是否满足, 调用结果是否和预期一致。

Observation 主要验证工具调用后的结果是否被准确记录, 包括成功数据、失败状态、错误码、超时、空结果和业务异常。

工具失败后最终回答不能说“已分析完成”或给出成功结论, 应该说明失败原因、缺失信息或建议用户重试。测试时我会用 TraceID 串 Tool Call 日志、Observation 和最终回答, 看三者是否一致。

### 问题 17: 结果是否足够回答这个判断怎么测

面试官问题:

“结果是否足够回答”这个判断怎么测? 什么情况下应该继续规划, 什么情况下应该终止?

可以这样答:

我会看工具结果和 RAG 证据是否已经覆盖用户目标。如果证据完整、工具成功、关键字段齐全, 就可以汇总回答。

如果缺关键证据、工具失败、参数不足、RAG 未命中或返回结果冲突, 就不能给确定结论, 应该终止并返回原因, 或提示用户补充信息。

如果本轮内部还有明确下一步, 比如一个工具返回了可继续查询的 ID, 且权限和参数都满足, 可以继续规划下一步。

用例上我会覆盖工具返回部分字段、空结果、错误码、超时、冲突结果、RAG 证据不足等情况, 看系统是否错误地把不足结果当成充分结果。

### 问题 18: 最终回答怎么评估

面试官问题:

最终回答怎么评估? 非确定性输出怎么断言?

可以这样答:

最终回答不能只看文案是否通顺, 要看答案、原因、下一步建议是否和工具结果、RAG 依据、权限状态一致。

对于非确定性输出, 我不会只做字符串完全匹配, 而是按事实点、业务规则、证据引用和风险等级来断言。比如日志分析类回答, 关键错误、故障方向、处理建议不能错; RAG 类回答, 制度、流程、接口字段必须有依据; 权限类回答不能泄露无权限内容。

同时我会沉淀标准样本和 badcase 样本, 用于 Prompt 变更、工具变更和知识库更新后的回归。

### 问题 19: 幻觉怎么测试

面试官问题:

幻觉怎么测试? 回答很流畅但没有依据怎么办?

可以这样答:

我会用无知识命中、缺附件、缺参数、工具失败、权限不足、冲突证据等样本来测试幻觉风险。

期望系统在没有依据时能澄清、拒答或说明无法确认, 而不是编造制度、流程、工具结果或分析结论。

如果回答很流畅, 但 RAG 未命中、工具未调用、Observation 失败或权限不足, 仍然给了确定结论, 我会定性为幻觉风险。定位时会记录输入、TraceID、RAG 结果、Tool Call、Observation 和最终回答, 形成 badcase 进入回归集。

### 问题 20: 线上 AI 质量问题怎么定位

面试官问题:

线上 AI 质量问题怎么定位? 怎么拆客户端、接口、RAG、工具和模型问题?

可以这样答:

我会先收集用户问题、截图、时间、账号、会话 ID、客户端版本、入口和 TraceID。

然后按链路排查: 客户端请求和附件上传是否正常, 接入层是否收到消息, 鉴权和意图分流是否正确, RAG 是否召回正确证据, Prompt/State 是否注入正确上下文, Tool Call 是否成功, Observation 是否记录正确, 最终回答是否基于证据生成。

如果客户端请求参数错、附件没传上来或页面展示错, 偏客户端问题。如果接口返回异常或接入层没有收到消息, 偏接口或接入问题。如果 RAG 证据错或没召回, 偏检索问题。如果工具结果错, 偏工具或下游服务问题。如果工具结果正确但回答错, 偏模型总结或 Prompt 问题。如果 Observation 失败但最终回答成功, 是结果判断或幻觉问题。

### 问题 21: Prompt 变更后怎么回归

面试官问题:

Prompt 变更后怎么回归? 新版流程下重点回归哪些节点?

可以这样答:

Prompt 变更后要重点回归缺参澄清、RAG 未命中、权限拒答、工具选择、参数校验、工具失败和最终回答格式。

因为 Prompt 不只影响表达, 还可能影响是否调用工具、是否编造答案、是否利用 RAG 证据、是否遵守 Observation 和权限边界。

我会用核心正常样本、历史 badcase、高频线上问题、权限边界、RAG 未命中、工具失败样本做分层回归。不只测正常问答, 还要测异常、拒答、澄清和失败处理。

### 问题 22: 工具 Schema 变更后怎么回归

面试官问题:

工具 Schema 变更后怎么回归? 参数名或枚举变了会影响什么?

可以这样答:

工具 Schema 变更会影响 Tool Selection、参数组装、参数校验和 Tool Call。

我会检查必填参数、参数名、类型、枚举、默认值、取值范围、错误返回是否和 Agent 侧适配。

如果参数名变了但 Agent 仍按旧字段传, 工具可能直接失败。如果枚举值变了, 可能调用成功但业务结果错误。如果必填项新增但 Prompt 或参数提取没有适配, 可能出现缺参仍调用。

回归时我会看 Tool Call 入参、工具返回、Observation 和最终回答, 确认链路一致。

### 问题 23: 元宝客户端和这个 Agent 项目的相似点是什么

面试官问题:

元宝客户端和这个项目的相似点是什么? 你如何把这个项目经验迁移到元宝?

可以这样答:

相似点是它们都不是单纯测客户端页面, 核心都是用户输入到 AI 回答的完整链路质量。

元宝有客户端入口、会话、文件或图片、多模态输入、RAG、Prompt、工具或插件能力、线上反馈。我的项目里也涉及用户输入、鉴权、意图识别、RAG、工具调用、Observation 和最终回答。

所以我的经验可以迁移到链路拆解、TraceID 定位、RAG 证据验证、权限边界、工具调用、异常返回和回归样本沉淀。客户端侧我会重点补充输入、附件、流式回答、历史会话、弱网、性能和稳定性这些场景。

### 问题 24: 如果面试官问你不是算法背景, 怎么回答

如果面试官问你不是算法背景, 你怎么回答? 你能做 AI 产品质量体系吗?

我会明确说我不是算法研发, 但我是 AI 应用测试和质量保障角色。

我不负责模型训练, 但我理解并测试过 Agent 应用链路里的意图分流、RAG、Prompt/State、工具调用、Observation、权限边界、幻觉和线上 badcase。

所以我可以做 AI 产品质量体系中的应用层质量保障, 重点是链路可观测、问题可定位、样本可回归。

### 问题 25: 元宝这类 AI 客户端核心功能测试点有哪些

面试官问题:

元宝这类 AI 客户端, 核心功能测试点有哪些?

可以这样答:

我会从用户核心使用链路覆盖: 登录和账号状态、会话创建、文本输入、图片或文件上传、发送问题、流式回答、停止生成、重新生成、复制分享、历史记录、会话删除、上下文连续追问、异常提示等。

同时要验证客户端展示和服务端返回一致, 比如 Markdown、代码块、表格、图片、链接、引用内容是否展示正确。

AI 客户端还要重点看输入和输出边界, 比如空输入、超长文本、特殊字符、重复发送、多轮追问、敏感内容拦截、回答中断和恢复。

### 问题 26: 流式回答怎么测

面试官问题:

流式回答怎么测? 包括首包、连续输出、中断、重试、重复内容、格式错乱。

可以这样答:

流式回答我会从功能、性能和异常三个方向测。

功能上看首包是否正常出现, 内容是否按顺序连续输出, Markdown、代码块、表格等格式在输出过程中和输出完成后是否正确, 是否出现重复内容、丢字、乱序或闪烁。

性能上关注首包时间、整体生成耗时、长回答输出过程中的卡顿和页面滚动体验。

异常上会测生成中断、网络断开、切后台、锁屏恢复、手动停止生成、重新生成, 看客户端状态是否正确, 是否支持重试, 是否重复发送请求。

定位时要结合客户端日志、接口状态、流式事件和 TraceID 判断问题在客户端渲染、网络链路还是服务端生成。

### 问题 27: 文件 / 图片上传怎么测

面试官问题:

文件/图片上传怎么测? 包括格式、大小、取消上传、上传失败、上传后提问、权限和异常提示。

可以这样答:

文件和图片上传我会先覆盖格式、大小、数量、文件名、损坏文件、空文件、超大文件、重复上传、取消上传、上传中断和上传失败。

上传成功后要验证文件 ID 或图片 ID 是否正确传到后端, 后续提问是否基于当前附件, 不会串到历史附件。

权限方面要看相册、文件、相机权限被拒绝时是否有正确提示, 不同系统版本的权限弹窗是否正常。

异常方面要覆盖弱网、断网、服务端超时、上传成功但解析失败、上传后立刻发送问题等场景。期望客户端状态清晰, 不重复上传, 不丢附件, 不生成无依据回答。

### 问题 28: 历史会话怎么测

面试官问题:

历史会话怎么测? 包括新建会话、切换会话、删除会话、多端同步、历史上下文污染。

可以这样答:

历史会话我会覆盖新建、重命名、切换、删除、搜索、加载更多、多端同步和离线恢复。

重点看切换会话后上下文是否正确, A 会话的附件、问题、权限状态不能带到 B 会话。删除会话后本地和服务端状态要一致, 已删除会话不能继续被追问引用。

多端同步要验证手机端、Web 端或其他端的会话列表、消息内容、删除状态是否一致。

还要覆盖长历史会话加载、网络异常下切换会话、历史消息渲染格式、重复消息和顺序错乱。

### 问题 29: 弱网场景怎么测

面试官问题:

弱网场景怎么测? 比如断网、网络抖动、接口超时、流式输出中断、恢复后是否重复发送。

可以这样答:

弱网我会覆盖断网、弱网、高延迟、丢包、网络抖动、Wi-Fi 和蜂窝网络切换、接口超时、服务端 5xx。

核心验证点是客户端状态是否清晰, 比如发送中、上传中、生成中、失败、可重试这些状态不能混乱。

流式回答中断时, 要看是否提示失败, 是否保留已生成内容, 重试时是否重复发送或生成重复内容。

附件上传弱网时, 要看是否出现上传成功但实际文件不可用、重复上传、附件 ID 丢失等问题。

定位时结合客户端日志、接口返回码、TraceID 和服务端日志判断是客户端重试策略、网络问题还是服务端异常。

### 问题 30: 客户端性能怎么测

面试官问题:

客户端性能怎么测? 包括启动、页面加载、首包时间、流式响应、内存、卡顿、耗电。

可以这样答:

客户端性能我会关注启动耗时、首页加载耗时、会话页加载耗时、发送请求后的首包时间、流式输出速度、长回答渲染性能、图片或文件上传耗时。

资源方面关注内存、CPU、卡顿、耗电和发热, 特别是长时间对话、连续生成、多图片上传、大文件上传这些场景。

测试时会区分客户端性能和服务端耗时, 比如首包慢可能是服务端生成慢, 但流式渲染卡顿可能是客户端 UI 渲染问题。

指标上可以结合客户端性能工具、埋点、接口耗时和 TraceID 做定位。

### 问题 31: 稳定性怎么测

面试官问题:

稳定性怎么测? 包括长时间对话、连续提问、后台切换、锁屏恢复、崩溃、ANR。

可以这样答:

稳定性我会覆盖长时间运行、多轮连续提问、长回答生成、大文件上传、频繁切后台、锁屏恢复、横竖屏切换、低电量、低内存和网络切换。

重点看是否出现崩溃、ANR、卡死、白屏、会话丢失、重复发送、生成状态无法结束。

对于 AI 客户端, 还要关注长会话下上下文是否错乱, 历史消息是否加载异常, 流式回答是否中断后无法恢复。

验证时会结合崩溃日志、ANR 日志、客户端日志、设备信息、版本号和 TraceID 定位。

### 问题 32: 线上客户端问题怎么定位

面试官问题:

线上用户反馈“回答没出来 / 卡住 / 内容重复 / 上传失败”, 你怎么定位是客户端问题还是服务端问题?

可以这样答:

我会先收集用户截图、时间、账号、客户端版本、设备型号、系统版本、网络环境、会话 ID 和 TraceID。

如果是回答没出来或卡住, 先看客户端是否成功发起请求, 接口是否返回, 流式事件是否到达客户端。如果服务端已经返回但页面没展示, 偏客户端渲染或状态管理问题。如果服务端没有返回或超时, 偏服务端或网络问题。

如果是内容重复, 要看客户端是否重复发送请求、服务端是否重复推送流式片段、客户端是否重复渲染。

如果是上传失败, 要看本地权限、文件大小和格式、上传接口状态、文件 ID 是否生成、后续请求是否带上附件 ID。

定位时用客户端日志、接口日志、TraceID、埋点和服务端日志串起来, 判断问题发生在哪一段。
