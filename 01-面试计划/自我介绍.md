# 自我介绍

你好, 我叫宋克宇, 目前在武汉, 有 7 年多软件测试和测试开发经验。我的经历覆盖服务端接口测试、Web/APP 测试、IoT 软硬件测试、性能测试、自动化测试和 AI 应用测试。技术栈方面, 我主要使用 Python、Pytest、Unittest 做接口自动化和回归测试, 使用 Locust 做接口性能和稳定性测试, 也熟悉 Jenkins、Git、Linux、MySQL、PostgreSQL、Postman、Wireshark、串口日志等工具。

工作经历上, 我 2019 年到 2022 年在武汉佰钧成参与阿里线项目, 做过天猫精灵 Mesh/Wi-Fi 芯片软件测试、生活物联网平台服务端接口测试, 也参与过 TMS 移动端自动化框架相关工作。那段经历让我比较熟悉接口自动化、软硬件联调、抓包、串口日志和复杂链路问题定位。2022 年至今, 我在北京外企德科参与华为 OD 测试工作, 主要负责生产维护数据分析平台、现网维护数据分析平台, 以及睿思 AI Agent 数字员工项目的测试工作。

最近最核心的项目是睿思 AI Agent 数字员工。这个项目面向光电维护场景, 用户会从 WeLink、Web 或 API 入口提问, 有时会上传日志或分析件。请求进入后会经过 UniKnow 做统一接入和账号绑定, 请求入口会保留消息、附件和 TraceID, 然后进入权限校验, 校验身份、工具权限和知识库权限。通过后由 Race Agent Service 做统一编排、上下文管理和路由, 再通过意图识别判断是普通问答、固定流程, 还是需要进入 Agent 的复杂任务。

如果进入 Agent, 系统会先判断目标、参数、附件和权限是否足够; 不足时会返回缺失参数说明, 本轮结束。上下文足够后再判断是否需要 RAG。需要依赖企业知识、规范、流程、接口文档、产品说明、内部字段或工具参数时, 会走 Query Rewrite、权限过滤、Retrieve/TopK、ReRank, 命中可用证据后注入 RAG 上下文; 未命中则标记知识不足或不注入 RAG。随后系统构建 Prompt/State, 进入 Agent Executor, 按 Plan / Thought / Action / Observation 执行, 再进行 Tool Selection、参数组装与校验、API/MCP/工作流/子 Agent 调用, 最后根据 Observation 判断结果是否足够回答。

我在这个项目里是测试人员角色, 主要介入需求评审、测试方案设计、环境验证、功能测试、集成测试、AI 效果评测、上线验收和迭代回归。测试时我不会只看最终回答, 而是按链路拆验证: TraceID 能否追踪, 权限失败是否被拦截, 意图分流是否正确, 缺参是否明确澄清, RAG 是否正确触发, Query Rewrite 是否改偏, 权限过滤是否生效, TopK/ReRank 证据是否可用, Prompt/State 是否包含必要上下文, 工具选择和参数组装是否正确, 工具失败时是否返回失败原因而不是编造成功结果。对 Prompt、RAG、工具 Schema、权限策略变更, 我会用历史缺陷、异常日志、高频问答和权限边界样本做回归评测。
