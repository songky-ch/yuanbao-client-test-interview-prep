# 自我介绍

面试官你好, 我叫宋克宇, 2018 年毕业于内蒙古科技大学, 计算机科学与技术专业。毕业后一直从事软件测试相关工作, 做过 IoT 软硬件测试、服务端接口测试、自动化测试、性能测试, 以及现在的 AI Agent 应用测试。2022 年开始, 我通过 OD 项目参与华为半导体维护开发部相关测试工作, 目前还在职。

我现在主要负责三个面向不同硬件部门用户的内部 Web 应用平台, 日常工作包括需求评审、测试方案设计、功能测试、接口测试、缺陷跟踪、上线验收和版本回归。这些平台主要支撑硬件部门的数据分析、维护分析和流程处理类工作, 所以我平时会比较关注数据准确性、权限控制、接口链路、异常分支和线上问题定位。

2025 年中之后, 我们团队开始做一个叫“睿小思”的数字员工 Agent 项目。这个项目的背景是这样的: 早期硬件部门有很多数据分析和处理工作需要人工完成, 比较耗人力, 所以我们组先做了一批 GUI 工具来支撑他们的工作。后来工具越来越多, 每个工具又比较定制化, 管理和使用成本都变高了。再往后我们尝试把这些能力云化, 最开始也是接入 WeLink, 但当时更像固定指令或固定格式的工作流。实际使用中, 因为工具名称相似、命令格式固定、用户同步不充分, 经常会出现用户用错命令、参数填错或者不知道该调用哪个能力的问题。

后来公司内部有一个平台提供了创建 Agent 的通用能力, 可以通过低代码配置 Agent、工具和知识库。我们就把原来数字员工相关能力接入进去, 把早期 GUI 工具逐步封装成 Tool, 让用户不用记固定命令, 而是通过自然语言在 WeLink 里发消息, 由 Agent 判断意图、提取参数、选择工具并返回结果。

现在这个 Agent 的主流程大概是这样: 用户通过 WeLink 发送指令消息, 如果涉及文件, 会引用文件一起发送。消息会先经过 UniKnow 平台, 这个平台主要做 WeLink 消息转发, 在这里可以看到用户原始输入、最终回复内容, 也可以通过 TraceID 查看日志来定位问题。UniKnow 转发后, 请求会进入我们自己的鉴权和意图分流服务, 先判断用户是否有使用这个 Agent 的权限, 以及是否具备对应工具或知识库的访问权限。

鉴权通过后, 会做意图分流, 判断用户的问题属于闲聊、文档问答, 还是工具调用/任务执行。如果是闲聊, 一般不进入 Agent 内部复杂流程, 直接走模型通用能力。如果是文档问答, 会走 RAG 知识库, 包括检索、权限过滤、召回和重排, 然后把命中的文档内容交给模型总结后返回。如果是工具调用或任务执行, 就会进入 Agent 内部流程。模型会先根据用户输入提取意图和参数, 判断需要调用哪个工具; 工具确定后, 再把提取出来的参数传给对应工具 API; 工具执行返回结果后, 再经过结果总结模型整理成用户能看懂的回复。

我在这个项目里是测试人员角色, 主要参与需求评审、测试方案设计、功能测试、集成测试、上线验收和迭代回归。测试重点不是只看最后回复对不对, 而是把整个链路拆开验证。比如 UniKnow 转发后用户原始输入是否完整, TraceID 是否能串起日志; 鉴权是否能拦住无权限用户; 意图分流是否能区分闲聊、文档问答和工具调用; RAG 场景下知识库召回是否准确, 是否存在无权限知识泄露; 工具调用场景下工具选择是否准确, 参数提取是否完整, 参数类型和业务域是否正确; 工具执行失败或缺参数时, Agent 是否会明确提示, 而不是编造一个成功结果。

因为这个 Agent 采用的是 ReAct 范式, 所以我也会关注它在计划、行动、观察结果这些环节上的表现。不过 Agent 内部推理过程并不是完全白盒, 所以我主要基于可观测证据来测试, 比如用户输入、TraceID 日志、意图分流结果、RAG 召回片段、工具调用记录、工具入参、工具返回结果、Observation 和最终回复之间是否一致。

举个例子, 如果用户上传文件后让 Agent 做分析, 我会看它是否识别到这是工具调用任务, 是否选择了正确工具, 是否把文件 ID、业务类型、用户权限等参数传给工具。如果工具返回失败, 最终回答就不能说“分析完成”; 如果 RAG 没有召回到有效知识, 模型也不能编造制度或流程说明。针对这些场景, 我们会沉淀回归样本集, 覆盖闲聊、文档问答、工具调用、缺参、无权限、工具失败、RAG 未命中、模型总结幻觉等场景, 用于后续 Prompt、知识库、工具配置或流程变更后的回归验证。

所以我这段项目经验和元宝客户端测试岗位比较贴近。元宝虽然是面向 C 端或更广泛用户的 AI 客户端, 但本质上也需要测试从用户输入、客户端交互、接口链路、RAG、Prompt、工具调用到最终回答的全链路质量。我的优势是既有传统功能、接口、日志定位和回归测试经验, 也实际参与过 AI Agent 应用链路的测试, 能从测试人员角度把 AI 输出问题拆到具体链路节点上定位。

## 自我介绍后高频追问

### Q1. 你们怎么判断用户问题是闲聊、文档问答还是工具调用?

参考答案: 我们这块是先经过 UniKnow 转发到我们自己的鉴权和意图分流服务。鉴权通过后, 会先做意图分流, 大概分三类: 闲聊、文档问答、工具调用或任务执行。

比如用户只是问一些通用问题, 或者不涉及我们内部业务工具和知识库, 一般就判断为闲聊, 不进入 Agent 内部复杂流程。如果用户问的是制度、流程、工具说明、字段含义、操作文档这类问题, 就会走文档问答, 也就是 RAG 知识库。如果用户的问题里明显包含“帮我分析这个文件”“查询某个结果”“执行某个分析任务”这类意图, 或者引用了文件, 就会走工具调用或任务执行。

我测试的时候不会只看最终回答, 会看分流结果和日志。比如同一个问题, 如果应该走文档问答但没有触发 RAG, 那可能就是意图分流漏判; 如果只是问文档说明却走了工具调用, 那就是误触发。我们会把闲聊、文档问答、工具调用、文件分析、模糊表达这些都放进回归样本里测。

### Q2. 工具名称相似时, 怎么验证 Agent 没选错工具?

参考答案: 这个场景我们项目里确实比较容易出问题, 因为早期 GUI 工具比较多, 有些工具名称和功能很接近。测试时我会先把这些相似工具整理成一组样本, 每个工具明确它适用的业务场景、必填参数、输入文件类型和期望输出。

比如两个工具都和数据分析有关, 但一个是做文件解析, 一个是做结果汇总, 那我会设计相似表达的问题, 看模型到底选择了哪个 Tool。验证时主要看三点: 第一是 Tool Selection 日志里选的工具是否正确; 第二是参数是否和这个工具的 Schema 匹配; 第三是工具返回结果和最终总结是否符合这个工具的业务含义。

如果最终回答看起来正常, 但调用的是另一个相似工具, 这个也要算缺陷, 因为结果可能只是“碰巧能返回”, 但业务链路是错的。

### Q3. 参数提取错误你们怎么发现?

参考答案: 参数提取我一般不会只看最终回答, 主要看工具调用前的入参。因为用户自然语言里可能会带文件、业务类型、时间范围、设备类型、分析目标这些信息, 模型要先把这些提取出来, 再传给对应工具 API。

测试时我会设计几类样本: 参数完整的、缺少参数的、参数名称相近的、口语化表达的、引用文件的、上下文里带参数的。然后对比用户原始输入、模型提取出来的参数、工具 API 实际收到的参数。

比如用户引用了一个文件让 Agent 分析, 那工具入参里应该有文件 ID 或文件标识; 如果用户说的是 A 业务域, 结果入参传成 B 业务域, 那就是参数提取或上下文取参错误。还有一种情况是参数缺失, 正确做法应该是提示用户补充, 而不是模型自己猜一个参数继续调用工具。

### Q4. RAG 召回错和模型总结错怎么区分?

参考答案: 我一般会把这个问题拆成两步看。第一步看 RAG 召回结果, 第二步看模型最终总结。

如果用户问的是文档、流程、字段说明这类问题, 我会先看知识库有没有召回到正确片段。如果召回片段本身就是错的, 或者完全没有命中相关内容, 那问题更偏 RAG 召回, 可能是 Query Rewrite、权限过滤、TopK 或 ReRank 的问题。

如果召回片段是对的, 但是最终回答总结错了, 比如漏掉关键限制条件, 或者把文档里的条件说反了, 那问题就更偏模型总结或 Prompt 约束。

所以我测试 RAG 时不会只看最终回答, 会看原始问题、改写后的 query、召回片段、重排结果和最终回复。这样才能判断到底是“没找到正确知识”, 还是“找到了但模型没用对”。

### Q5. UniKnow 上能看到哪些日志? TraceID 怎么串链路?

参考答案: UniKnow 这边主要能看到用户从 WeLink 发过来的原始文本、引用文件信息、最终返回给用户的回复内容, 以及对应的 TraceID。这个 TraceID 对测试定位很重要, 因为它可以把一次用户请求从入口一路串到后面的服务日志。

比如用户反馈回答不对, 我会先在 UniKnow 上根据时间、用户和 TraceID 找到原始输入, 确认用户到底发了什么, 有没有引用文件, 最终返回是什么。然后再拿 TraceID 去查我们自己的鉴权和意图分流日志, 看用户有没有权限、分流到了闲聊还是 RAG 还是工具调用。

如果是工具调用, 继续查工具选择、参数提取、工具 API 入参和工具返回结果。这样就能判断问题是在 UniKnow 转发、权限、意图分流、RAG、工具调用, 还是最后模型总结。

### Q6. 工具调用失败后, 你怎么判断最终回答有没有幻觉?

参考答案: 工具调用失败后, 我主要看最终回答和工具返回结果是否一致。正常情况下, 如果工具失败、超时、参数缺失, 最终回答应该明确说明失败原因或提示用户补充信息, 不能说“分析完成”或者给出一个看起来很确定的结论。

测试时我会故意构造一些异常场景, 比如工具必填参数缺失、文件解析失败、工具接口返回错误码、工具超时、用户无权限调用工具。然后看 Observation 或工具返回日志是什么, 再对比最终回复。

如果工具日志里明确是失败, 但最终回答说“已完成分析, 结果是……”, 那就是典型幻觉或结果总结不可信。这个问题风险比较高, 因为用户会以为工具真的执行成功了。

### Q7. 你们回归样本集怎么设计?

参考答案: 我们回归样本集不是只放正常问题, 而是按 Agent 链路来分层设计。

第一类是闲聊和普通问答, 用来看不会误进复杂 Agent 流程。第二类是文档问答, 覆盖制度、流程、字段说明、工具说明, 重点看 RAG 召回和模型总结。第三类是工具调用, 覆盖每个核心工具的正常调用、相似工具选择、参数提取、文件引用和工具返回结果。第四类是异常场景, 比如缺参数、无权限、文件缺失、工具失败、RAG 未命中。第五类是历史缺陷样本, 之前出过问题的表达方式、工具误选、参数提取错误、总结幻觉都会沉淀进去。

每次 Prompt、知识库、工具配置或者意图分流逻辑有变更, 我们就用这些样本做回归。重点不是跑一遍有回答就行, 而是看分流、RAG、工具调用、参数和最终回答有没有回退。

### Q8. 你作为测试人员, 在 Agent 项目里最有价值的工作是什么?

参考答案: 我觉得最有价值的不是单纯点页面或者看最后回答, 而是把 Agent 这个黑盒回答拆成可以验证、可以定位、可以回归的链路。

因为用户看到的只是 WeLink 里的一段回复, 但这个回复背后可能经过 UniKnow 转发、鉴权、意图分流、RAG、工具选择、参数提取、工具 API、结果总结。测试如果只说“回答不对”, 研发和算法其实很难定位。

所以我在项目里做的事情, 一方面是从需求评审阶段就把风险点拆出来, 比如权限、工具误选、参数缺失、RAG 召回错、工具失败后幻觉。另一方面是测试执行时通过 TraceID、日志、工具入参、RAG 召回片段、Observation 和最终回答去定位问题。最后再把典型问题沉淀成回归样本, 后面版本迭代时能快速验证有没有回退。

这样测试的价值就不是只发现 bug, 而是帮助团队把 Agent 的质量变得可控。
