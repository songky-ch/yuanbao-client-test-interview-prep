# 3 天缓冲版

## 使用目标

在 1 天紧急版基础上, 按新版流程图逐层深挖。训练目标是: 面试官问到任何节点, 都能从测试人员角度回答测试设计、定位证据、缺陷风险和回归方法。

## 定稿必背: 自我介绍与项目介绍

面试官你好, 我叫宋克宇, 2018 年毕业于内蒙古科技大学, 计算机科学与技术专业。毕业后一直从事软件测试相关工作, 做过 IoT 软硬件测试、服务端接口测试、自动化测试、性能测试, 以及现在的 AI Agent 应用测试。2022 年开始, 我通过 OD 项目参与华为半导体维护开发部相关测试工作, 目前还在职。

我现在主要负责三个面向不同硬件部门用户的内部 Web 应用平台, 日常工作包括需求评审、测试方案设计、功能测试、接口测试、缺陷跟踪、上线验收和版本回归。这些平台主要支撑硬件部门的数据分析、维护分析和流程处理类工作, 所以我平时会比较关注数据准确性、权限控制、接口链路、异常分支和线上问题定位。

2025 年中之后, 我们团队开始做一个叫“睿小思”的数字员工 Agent 项目。这个项目的背景是这样的: 早期硬件部门有很多数据分析和处理工作需要人工完成, 比较耗人力, 所以我们组先做了一批 GUI 工具来支撑他们的工作。后来工具越来越多, 每个工具又比较定制化, 管理和使用成本都变高了。再往后我们尝试把这些能力云化, 最开始也是接入 WeLink, 但当时更像固定指令或固定格式的工作流。实际使用中, 因为工具名称相似、命令格式固定、用户同步不充分, 经常会出现用户用错命令、参数填错或者不知道该调用哪个能力的问题。

后来公司内部有一个平台提供了创建 Agent 的通用能力, 可以通过低代码配置 Agent、工具和知识库。我们就把原来数字员工相关能力接入进去, 把早期 GUI 工具逐步封装成 Tool, 让用户不用记固定命令, 而是通过自然语言在 WeLink 里发消息, 由 Agent 判断意图、提取参数、选择工具并返回结果。

现在这个 Agent 的主流程大概是这样: 用户通过 WeLink 发送指令消息, 如果涉及文件, 会引用文件一起发送。消息会先经过 UniKnow 平台, 这个平台主要做 WeLink 消息转发, 在这里可以看到用户原始输入、最终回复内容, 也可以通过 TraceID 查看日志来定位问题。UniKnow 转发后, 请求会进入我们自己的鉴权和意图分流服务, 先判断用户是否有使用这个 Agent 的权限, 以及是否具备对应工具或知识库的访问权限。

鉴权通过后, 会做意图分流, 判断用户的问题属于闲聊、文档问答, 还是工具调用/任务执行。如果是闲聊, 一般不进入 Agent 内部复杂流程, 直接走模型通用能力。如果是文档问答, 会走 RAG 知识库, 包括检索、权限过滤、召回和重排, 然后把命中的文档内容交给模型总结后返回。如果是工具调用或任务执行, 就会进入 Agent 内部流程。模型会先根据用户输入提取意图和参数, 判断需要调用哪个工具; 工具确定后, 再把提取出来的参数传给对应工具 API; 工具执行返回结果后, 再经过结果总结模型整理成用户能看懂的回复。

我在这个项目里是测试人员角色, 主要参与需求评审、测试方案设计、功能测试、集成测试、上线验收和迭代回归。测试重点不是只看最后回复对不对, 而是把整个链路拆开验证。比如 UniKnow 转发后用户原始输入是否完整, TraceID 是否能串起日志; 鉴权是否能拦住无权限用户; 意图分流是否能区分闲聊、文档问答和工具调用; RAG 场景下知识库召回是否准确, 是否存在无权限知识泄露; 工具调用场景下工具选择是否准确, 参数提取是否完整, 参数类型和业务域是否正确; 工具执行失败或缺参数时, Agent 是否会明确提示, 而不是编造一个成功结果。

因为这个 Agent 采用的是 ReAct 范式, 所以我也会关注它在计划、行动、观察结果这些环节上的表现。不过 Agent 内部推理过程并不是完全白盒, 所以我主要基于可观测证据来测试, 比如用户输入、TraceID 日志、意图分流结果、RAG 召回片段、工具调用记录、工具入参、工具返回结果、Observation 和最终回复之间是否一致。

举个例子, 如果用户上传文件后让 Agent 做分析, 我会看它是否识别到这是工具调用任务, 是否选择了正确工具, 是否把文件 ID、业务类型、用户权限等参数传给工具。如果工具返回失败, 最终回答就不能说“分析完成”; 如果 RAG 没有召回到有效知识, 模型也不能编造制度或流程说明。针对这些场景, 我们会沉淀回归样本集, 覆盖闲聊、文档问答、工具调用、缺参、无权限、工具失败、RAG 未命中、模型总结幻觉等场景, 用于后续 Prompt、知识库、工具配置或流程变更后的回归验证。

所以我这段项目经验和元宝客户端测试岗位比较贴近。元宝虽然是面向 C 端或更广泛用户的 AI 客户端, 但本质上也需要测试从用户输入、客户端交互、接口链路、RAG、Prompt、工具调用到最终回答的全链路质量。我的优势是既有传统功能、接口、日志定位和回归测试经验, 也实际参与过 AI Agent 应用链路的测试, 能从测试人员角度把 AI 输出问题拆到具体链路节点上定位。

## 定稿必练: 自我介绍后追问

### Q0-1. 你们怎么判断用户问题是闲聊、文档问答还是工具调用?

参考答案: 我们这块是先经过 UniKnow 转发到我们自己的鉴权和意图分流服务。鉴权通过后, 会先做意图分流, 大概分三类: 闲聊、文档问答、工具调用或任务执行。

比如用户只是问一些通用问题, 或者不涉及我们内部业务工具和知识库, 一般就判断为闲聊, 不进入 Agent 内部复杂流程。如果用户问的是制度、流程、工具说明、字段含义、操作文档这类问题, 就会走文档问答, 也就是 RAG 知识库。如果用户的问题里明显包含“帮我分析这个文件”“查询某个结果”“执行某个分析任务”这类意图, 或者引用了文件, 就会走工具调用或任务执行。

我测试的时候不会只看最终回答, 会看分流结果和日志。比如同一个问题, 如果应该走文档问答但没有触发 RAG, 那可能就是意图分流漏判; 如果只是问文档说明却走了工具调用, 那就是误触发。我们会把闲聊、文档问答、工具调用、文件分析、模糊表达这些都放进回归样本里测。

### Q0-2. 工具名称相似时, 怎么验证 Agent 没选错工具?

参考答案: 这个场景我们项目里确实比较容易出问题, 因为早期 GUI 工具比较多, 有些工具名称和功能很接近。测试时我会先把这些相似工具整理成一组样本, 每个工具明确它适用的业务场景、必填参数、输入文件类型和期望输出。

比如两个工具都和数据分析有关, 但一个是做文件解析, 一个是做结果汇总, 那我会设计相似表达的问题, 看模型到底选择了哪个 Tool。验证时主要看三点: 第一是 Tool Selection 日志里选的工具是否正确; 第二是参数是否和这个工具的 Schema 匹配; 第三是工具返回结果和最终总结是否符合这个工具的业务含义。

如果最终回答看起来正常, 但调用的是另一个相似工具, 这个也要算缺陷, 因为结果可能只是“碰巧能返回”, 但业务链路是错的。

### Q0-3. 参数提取错误你们怎么发现?

参考答案: 参数提取我一般不会只看最终回答, 主要看工具调用前的入参。因为用户自然语言里可能会带文件、业务类型、时间范围、设备类型、分析目标这些信息, 模型要先把这些提取出来, 再传给对应工具 API。

测试时我会设计几类样本: 参数完整的、缺少参数的、参数名称相近的、口语化表达的、引用文件的、上下文里带参数的。然后对比用户原始输入、模型提取出来的参数、工具 API 实际收到的参数。

比如用户引用了一个文件让 Agent 分析, 那工具入参里应该有文件 ID 或文件标识; 如果用户说的是 A 业务域, 结果入参传成 B 业务域, 那就是参数提取或上下文取参错误。还有一种情况是参数缺失, 正确做法应该是提示用户补充, 而不是模型自己猜一个参数继续调用工具。

### Q0-4. RAG 召回错和模型总结错怎么区分?

参考答案: 我一般会把这个问题拆成两步看。第一步看 RAG 召回结果, 第二步看模型最终总结。

如果用户问的是文档、流程、字段说明这类问题, 我会先看知识库有没有召回到正确片段。如果召回片段本身就是错的, 或者完全没有命中相关内容, 那问题更偏 RAG 召回, 可能是 Query Rewrite、权限过滤、TopK 或 ReRank 的问题。

如果召回片段是对的, 但是最终回答总结错了, 比如漏掉关键限制条件, 或者把文档里的条件说反了, 那问题就更偏模型总结或 Prompt 约束。

所以我测试 RAG 时不会只看最终回答, 会看原始问题、改写后的 query、召回片段、重排结果和最终回复。这样才能判断到底是“没找到正确知识”, 还是“找到了但模型没用对”。

### Q0-5. UniKnow 上能看到哪些日志? TraceID 怎么串链路?

参考答案: UniKnow 这边主要能看到用户从 WeLink 发过来的原始文本、引用文件信息、最终返回给用户的回复内容, 以及对应的 TraceID。这个 TraceID 对测试定位很重要, 因为它可以把一次用户请求从入口一路串到后面的服务日志。

比如用户反馈回答不对, 我会先在 UniKnow 上根据时间、用户和 TraceID 找到原始输入, 确认用户到底发了什么, 有没有引用文件, 最终返回是什么。然后再拿 TraceID 去查我们自己的鉴权和意图分流日志, 看用户有没有权限、分流到了闲聊还是 RAG 还是工具调用。

如果是工具调用, 继续查工具选择、参数提取、工具 API 入参和工具返回结果。这样就能判断问题是在 UniKnow 转发、权限、意图分流、RAG、工具调用, 还是最后模型总结。

### Q0-6. 工具调用失败后, 你怎么判断最终回答有没有幻觉?

参考答案: 工具调用失败后, 我主要看最终回答和工具返回结果是否一致。正常情况下, 如果工具失败、超时、参数缺失, 最终回答应该明确说明失败原因或提示用户补充信息, 不能说“分析完成”或者给出一个看起来很确定的结论。

测试时我会故意构造一些异常场景, 比如工具必填参数缺失、文件解析失败、工具接口返回错误码、工具超时、用户无权限调用工具。然后看 Observation 或工具返回日志是什么, 再对比最终回复。

如果工具日志里明确是失败, 但最终回答说“已完成分析, 结果是……”, 那就是典型幻觉或结果总结不可信。这个问题风险比较高, 因为用户会以为工具真的执行成功了。

### Q0-7. 你们回归样本集怎么设计?

参考答案: 我们回归样本集不是只放正常问题, 而是按 Agent 链路来分层设计。

第一类是闲聊和普通问答, 用来看不会误进复杂 Agent 流程。第二类是文档问答, 覆盖制度、流程、字段说明、工具说明, 重点看 RAG 召回和模型总结。第三类是工具调用, 覆盖每个核心工具的正常调用、相似工具选择、参数提取、文件引用和工具返回结果。第四类是异常场景, 比如缺参数、无权限、文件缺失、工具失败、RAG 未命中。第五类是历史缺陷样本, 之前出过问题的表达方式、工具误选、参数提取错误、总结幻觉都会沉淀进去。

每次 Prompt、知识库、工具配置或者意图分流逻辑有变更, 我们就用这些样本做回归。重点不是跑一遍有回答就行, 而是看分流、RAG、工具调用、参数和最终回答有没有回退。

### Q0-8. 你作为测试人员, 在 Agent 项目里最有价值的工作是什么?

参考答案: 我觉得最有价值的不是单纯点页面或者看最后回答, 而是把 Agent 这个黑盒回答拆成可以验证、可以定位、可以回归的链路。

因为用户看到的只是 WeLink 里的一段回复, 但这个回复背后可能经过 UniKnow 转发、鉴权、意图分流、RAG、工具选择、参数提取、工具 API、结果总结。测试如果只说“回答不对”, 研发和算法其实很难定位。

所以我在项目里做的事情, 一方面是从需求评审阶段就把风险点拆出来, 比如权限、工具误选、参数缺失、RAG 召回错、工具失败后幻觉。另一方面是测试执行时通过 TraceID、日志、工具入参、RAG 召回片段、Observation 和最终回答去定位问题。最后再把典型问题沉淀成回归样本, 后面版本迭代时能快速验证有没有回退。

这样测试的价值就不是只发现 bug, 而是帮助团队把 Agent 的质量变得可控。

## Day 1: 流程图逐节点深挖

### Q1. WeLink / Web / API 三类入口怎么测?

参考答案: 我会分别验证消息格式、附件上传、账号信息、TraceID、异常返回和入口差异。比如同一个问题从 Web 和 API 进入, 后端收到的用户身份、附件 ID、会话 ID 应一致。入口层问题不能归因到模型, 所以要先把接入参数和日志打通。

### Q2. UniKnow 的测试重点是什么?

参考答案: UniKnow 负责统一接入和账号绑定, 测试重点是账号绑定成功/失败、登录失效、域或组织信息是否正确传递、不同账号权限是否隔离。它的异常应在进入 Agent 前暴露, 不能让后续服务拿到错误身份继续执行。

### Q3. 请求入口里的消息、附件、TraceID 怎么设计用例?

参考答案: 消息测空内容、超长内容、特殊字符、模糊表达; 附件测格式、大小、空文件、损坏文件、解析失败; TraceID 测生成、透传、日志可查。真实定位时, TraceID 要能串起入口、权限、RAG、工具调用和最终回答。

### Q4. 鉴权失败应该怎么返回?

参考答案: 应明确返回无权限或登录失效, 不应进入 RAG、Agent Executor 或工具调用。测试时要确认鉴权失败后没有知识召回、没有 Tool Selection、没有 Tool Call。否则即使最终回答拒绝, 中间链路也可能已发生越权风险。

### Q5. 工具权限和知识库权限哪个风险更高?

参考答案: 两者都高。工具权限风险是越权调用真实业务能力; 知识库权限风险是泄露内部知识。测试时要分别验证: 无工具权限不能选工具, 无知识权限不能召回片段, 无权限内容不能进入 Prompt/State。

### Q6. 意图识别错误会造成什么后果?

参考答案: 普通问答误进 Agent 会增加耗时和误调用工具风险; 复杂任务没进 Agent 会回答泛化; 工具任务分错会导致参数和业务结果错误。我会用规则问答、固定流程、日志分析、实时查询、知识问答样本验证分流。

### Q7. “是否进入 Agent”这个分支怎么验收?

参考答案: 验收标准不是“有回答”, 而是路由正确。普通问答和固定流程应不进 Agent; 需要规划、工具调用、复杂判断的任务应进入 Agent。检查点包括路由日志、Agent Executor 是否启动、是否发生 Tool Selection。

### Q8. 上下文不足为什么本轮结束?

参考答案: 新版流程图定义单轮 Agent 缺参结束, 不等待用户补充后恢复执行。这能避免系统在参数不完整时猜测执行。测试时要验证缺参返回后没有继续 RAG、Prompt/State、Tool Call; 用户下一次补充信息时按新请求重新进入流程。

### Q9. 一次性澄清问题怎么测试?

参考答案: 我会准备多参数缺失样本, 看系统是否一次性说明缺哪些信息, 而不是只问一个字段导致用户反复补充。比如日志分析缺附件、业务域、分析目标时, 回答应明确列出缺失项。

### Q10. 哪些任务需要 RAG?

参考答案: 制度、规范、流程、接口文档、产品说明、内部字段、工具参数、业务概念映射类问题需要 RAG。实时接口查询、当前输入已足够的问题可以不需要 RAG。测试时要看是否触发 RAG, 以及未触发时回答是否仍有足够依据。

### Q11. Query Rewrite 的风险是什么?

参考答案: 最大风险是改写偏离用户原意。比如用户问内部字段 A 在某业务域的含义, 改写成泛化字段说明, 就会召回错误知识。我会对比原问题、改写 query、TopK 结果和最终回答, 判断是否保留关键实体、业务域和意图。

### Q12. 权限过滤放在 RAG 链路里怎么测?

参考答案: 用同一问题、不同部门账号测试。无权限账号不应召回对应知识, 更不能把片段注入 Prompt。测试证据包括召回日志、RAG 上下文和最终回答。如果最终回答没有泄露, 但日志显示无权限片段被注入, 也应判为风险。

### Q13. TopK 召回怎么评估?

参考答案: 我会构造标准答案已知的问题, 检查 TopK 是否包含正确片段, 正确片段排名是否靠前, 是否混入高干扰无关片段。TopK 评估要看召回率和相关性, 不能只看最终回答。

### Q14. ReRank 怎么评估?

参考答案: ReRank 要把最能支撑回答的证据排前。测试时准备多个相似片段、过期片段、冲突片段, 看重排后是否选择正确证据。如果 TopK 有正确片段但 ReRank 选错, 最终回答容易被错误证据带偏。

### Q15. RAG 命中和未命中分别怎么返回?

参考答案: 命中时应注入可用证据到 Prompt/State, 最终回答基于证据。未命中时, 如果问题依赖企业知识, 应标记知识不足、提示无法确认或要求补充资料; 如果问题不依赖知识库, 可以走普通执行。关键是不把未命中包装成确定答案。

### Q16. Prompt/State 构建有哪些测试点?

参考答案: 要看输入、附件解析结果、RAG 依据、权限状态、上下文是否完整进入 State。缺附件会导致日志分析泛化; 缺 RAG 会导致无依据回答; 缺权限会导致越权工具或知识使用。

### Q17. Plan / Thought / Action / Observation 里测试人员看什么?

参考答案: 看执行链路是否能解释最终结果。Plan 是否覆盖用户目标, Action 是否选择正确工具, Observation 是否记录工具结果或错误, 后续回答是否基于 Observation。测试人员不评价模型内部推理优雅不优雅, 重点看可观测、可定位、可复现。

### Q18. “下一步是否明确”怎么测?

参考答案: 我会构造工具、参数、目标、输出不明确的样本。明确时应进入 Tool Selection; 不明确时返回澄清并结束。比如用户没有给日志文件却要求分析, 下一步不明确, 不应调用工具。

### Q19. Tool Selection 按权限和 Schema 是什么意思?

参考答案: 权限决定能不能选工具, Schema 决定工具需要什么参数和格式。测试时要验证无权限不选工具、有权限才选工具、工具选择和任务类型匹配、Schema 必填项缺失时不调用。

### Q20. 参数组装为什么容易出问题?

参考答案: 参数可能来自用户输入、RAG、上下文和附件解析。任何来源错都会导致工具调用错。比如业务域从旧上下文取错、附件 ID 丢失、RAG 片段里的字段被误当参数, 都会导致错误结果。测试要看入参日志和校验结果。

### Q21. Observation 在定位中怎么用?

参考答案: Observation 是工具结果、错误、业务状态的证据。最终回答必须和 Observation 一致。若 Observation 是工具失败, 最终回答却说分析成功, 问题就在结果判断或生成阶段。线上定位时 Observation 能区分工具问题和回答总结问题。

### Q22. 结果是否足够回答怎么判断?

参考答案: 判断依据是工具结果和 RAG 证据是否覆盖用户目标。工具空结果、超时、错误码、证据冲突、参数缺失都不应生成确定结论。测试时要准备部分结果、失败结果、冲突结果, 看是否正确终止或继续规划。

### Q23. 异常/澄清返回有哪些场景?

参考答案: 缺参、无权限、工具失败、知识不足、结果不足。每类异常都应有明确原因, 不能只返回通用失败。比如“缺少日志附件”和“知识库未命中”对用户下一步动作不同, 需要区分。

### Q24. 单轮 Agent 约束对测试有什么影响?

参考答案: 缺参后本轮结束, 下一次输入按新请求进入流程。测试时不能按传统多轮任务恢复逻辑设计预期, 要验证系统不会把上一次未完成的执行状态错误延续到下一次请求。

### Q25. 这个流程图里最容易出线上问题的节点有哪些?

参考答案: 权限过滤、Query Rewrite、TopK/ReRank、参数组装、Tool Selection、Observation 到最终回答的一致性。因为这些节点出错时, 用户看到的可能只是一个“看起来正常”的回答, 但实际依据、权限或工具结果是错的。

## Day 2: 测试设计、自动化与定位

### Q26. 你会如何为这套流程设计测试用例?

参考答案: 按链路分层: 接入鉴权、意图分流、上下文完整性、RAG、Prompt/State、工具调用、结果返回。每层覆盖正常、异常、边界、权限、历史缺陷和回归样本。这样能定位问题归属, 不会只停留在最终回答。

### Q27. 如何设计 RAG 回归样本集?

参考答案: 分为制度规范、流程说明、接口文档、产品说明、内部字段、工具参数、无答案、无权限、冲突知识、过期知识。每条样本保留问题、账号权限、期望证据、期望回答要点、实际 TopK/ReRank 和最终回答。

### Q28. 如何设计工具调用回归样本集?

参考答案: 分为正确触发、不应触发、无权限、缺必填参数、参数类型错、工具超时、工具失败、Observation 为空、结果不足。每条样本要记录 Tool Selection、入参、Tool Call 结果、Observation、最终回答。

### Q29. 如何设计权限测试矩阵?

参考答案: 维度包括账号角色、部门、知识库权限、工具权限、登录状态、群组/组织关系。用同一问题在不同账号下执行, 对比召回结果、工具选择和最终回答。权限问题要看中间链路, 不能只看最终文本。

### Q30. 如何设计缺参测试矩阵?

参考答案: 按目标缺失、附件缺失、业务域缺失、工具参数缺失、权限缺失、输出要求缺失设计。期望是一次性说明缺失项并结束本轮, 不进入后续执行。

### Q31. 如何做 AI Agent 接口自动化?

参考答案: 可以把样本作为数据驱动, 调接口提交问题和附件, 记录 TraceID, 再拉取或解析日志中的 RAG、Tool Call、Observation 和最终回答。断言分规则断言和人工复核: 权限、是否触发工具、是否缺参返回可以自动断; 开放回答按关键事实点和风险等级判断。

### Q32. Pytest 在这个项目里可以怎么用?

参考答案: 用 Pytest 管理数据驱动样本、接口请求、环境配置、报告和回归执行。fixture 处理登录、账号权限、附件上传、TraceID 收集。参数化覆盖不同问题类型。断言包括状态码、业务码、RAG 是否触发、工具是否调用、最终回答关键点。

### Q33. Jenkins 回归失败怎么分析?

参考答案: 先看失败集中在哪类样本。如果权限样本集中失败, 查权限策略或账号配置; RAG 样本失败, 查知识库、Query Rewrite、TopK/ReRank; 工具样本失败, 查 Schema 或工具服务; 大面积失败, 查环境、鉴权或公共入口。

### Q34. 性能测试怎么覆盖这条链路?

参考答案: 分普通问答、RAG 问答、工具任务、附件日志分析几类压测, 不混成一个平均值。指标包括入口响应、首包时间、总耗时、RAG 耗时、工具耗时、错误率、超时率、资源使用。AI Agent 性能瓶颈可能在 RAG、工具或模型生成, 不一定在入口接口。

### Q35. 线上反馈“回答慢”怎么定位?

参考答案: 用 TraceID 拆耗时: 入口、鉴权、意图识别、Query Rewrite、TopK、ReRank、Prompt/State、Tool Call、Observation、最终生成。哪个阶段耗时高, 就定位到对应服务或依赖。

### Q36. 线上反馈“答非所问”怎么定位?

参考答案: 先看用户原问题和改写 query 是否偏离, 再看意图分流是否错、RAG 证据是否相关、Prompt/State 是否包含无关上下文、工具是否误选。答非所问通常不是单一模型问题, 要沿链路拆。

### Q37. 线上反馈“明明有知识但回答不知道”怎么定位?

参考答案: 查 RAG 是否触发、Query Rewrite 是否保留关键词、权限过滤是否误过滤、TopK 是否召回、ReRank 是否筛掉正确证据、RAG 上下文是否注入 Prompt。逐层定位是知识缺失、检索问题还是生成问题。

### Q38. 线上反馈“回答泄露了不该看的内容”怎么定位?

参考答案: 优先查权限链路: 身份是否正确、部门权限是否正确、知识库权限过滤是否生效、无权限片段是否进入 RAG 上下文、最终回答是否引用。越权问题优先级高, 需要立即复现和回归权限矩阵。

### Q39. 线上反馈“工具执行了但结果不对”怎么定位?

参考答案: 查 Tool Selection 是否正确、参数组装是否正确、Tool Call 是否成功、Observation 结果是什么、最终回答是否正确引用 Observation。如果工具返回本身错, 归工具或业务服务; 工具返回对但回答错, 归结果总结或 Prompt。

### Q40. 如何判断是客户端问题还是 Agent 后端问题?

参考答案: 看客户端请求和后端日志是否一致。客户端没传附件、会话 ID、账号信息, 是客户端或接入问题; 后端收到正确请求但 RAG、工具、回答错, 是 Agent 后端链路问题; 后端返回正确但页面展示错, 是客户端展示问题。

### Q41. 如何用日志证明一个 AI 缺陷?

参考答案: 最少要有用户输入、账号权限、TraceID、分流结果、RAG 结果、Tool Call/Observation、最终回答和期望规则。AI 缺陷不能只说“回答不好”, 要说明哪个节点错、错的证据是什么、影响范围是什么。

### Q42. 如何沉淀回归资产?

参考答案: 每个 badcase 记录输入、账号权限、附件、TraceID、触发节点、期望结果、实际结果、修复方案和复测结果。按权限、RAG、工具、缺参、幻觉、性能、客户端展示分类, 进入版本回归。

## Day 3: 岗位贴合与高压追问

### Q43. 元宝客户端测试为什么需要理解 AI Agent 流程?

参考答案: 因为 AI 客户端的问题经常表现为页面上的一个回答, 但根因可能在入口参数、权限、RAG、Prompt、工具调用、模型生成或前端展示。不了解链路, 只能报“回答不对”, 无法推动研发、算法、产品定位。

### Q44. 元宝的多模态能力和你项目里的附件/日志链路怎么迁移?

参考答案: 我项目里主要测日志和分析件上传, 思路可迁移到图片、文件、语音等多模态。重点是上传、解析、内容抽取、权限、上下文注入和最终回答。上传成功不等于 AI 拿到了正确内容, 必须验证预处理和 State。

### Q45. 如何制定 AI 产品发布准入标准?

参考答案: 准入标准包括核心样本通过率、严重幻觉为零、越权问题为零、工具调用高风险问题为零、历史缺陷不复现、RAG 核心知识命中、客户端核心链路稳定、性能指标达标。开放回答可人工复核重点样本。

### Q46. 如何搭建 AI 产品质量指标?

参考答案: 离线指标包括样本通过率、RAG 命中率、工具调用成功率、缺参澄清正确率、权限拦截正确率、历史缺陷复现率。线上指标包括点赞点踩、重试率、无答案率、转人工率、工具失败率、响应耗时、badcase 数量。

### Q47. 如何处理模型输出不确定性?

参考答案: 不用逐字匹配, 用事实点、业务规则、证据引用、风险等级判定。对高风险场景要求结论稳定; 对低风险表达可允许差异。必要时同一样本多次执行, 看关键结论是否一致。

### Q48. Prompt 变更为什么可能影响工具调用?

参考答案: Prompt 会影响模型如何理解任务、选择工具、生成参数和处理失败。如果工具描述或边界规则变了, 可能导致误选工具、漏选工具、缺参仍执行或工具失败后编造结果。因此 Prompt 变更要回归工具样本和异常样本。

### Q49. 工具 Schema 变更为什么是高风险?

参考答案: Schema 决定参数名、类型、必填项、枚举和返回结构。Schema 变更后, Agent 可能仍按旧参数调用, 导致工具失败或业务结果错。测试要看入参、校验、Tool Call 和 Observation, 不能只看最终回答。

### Q50. Query Rewrite 和 Prompt 优化有什么区别?

参考答案: Query Rewrite 影响检索输入, 决定召回什么知识; Prompt 优化影响模型如何使用输入、证据和工具结果。Rewrite 错会导致证据错, Prompt 错会导致证据对但回答错。定位时要先看检索证据, 再看生成。

### Q51. 如何测试提示词注入?

参考答案: 构造“忽略之前规则”“输出系统提示”“绕过权限”“调用某工具”等输入, 看是否影响权限、RAG、Tool Selection 和最终回答。Agent 场景下提示词注入不只是回答风险, 还可能造成越权工具调用。

### Q52. 如何判断一个问题是产品策略还是缺陷?

参考答案: 看需求和验收标准。如果流程图明确缺参要结束, 但系统继续执行, 是缺陷; 如果未命中知识时产品策略就是普通回答, 则按策略验收。测试人员要把现象、证据、风险和标准对齐, 不凭主观判断。

### Q53. 算法说“模型偶现正常波动”, 你怎么推进?

参考答案: 我会提供样本复现次数、关键事实错误比例、影响业务场景和风险等级。如果只是表达差异, 可以接受; 如果关键结论、权限、工具调用错误, 就不能用波动解释, 需要进入缺陷或策略优化。

### Q54. 产品要求快速上线, 你怎么压缩测试范围?

参考答案: 保权限、核心链路、历史缺陷、高风险幻觉、工具调用、缺参和失败返回。低风险 UI 文案或非核心体验可记录风险。AI 产品不能放过越权、错误指导、编造成功结果这类问题。

### Q55. 你如何和客户端、后端、算法、产品协作?

参考答案: 我会用 TraceID 和节点证据沟通。客户端看请求和展示, 后端看接入、权限、工具和日志, 算法看 RAG、Prompt、生成, 产品确认策略和验收标准。测试负责把问题拆清楚, 而不是只把“回答不对”丢给某一方。

### Q56. 你简历里 Python/Pytest 能如何支撑 AI 测试?

参考答案: 可以把 AI 样本做成数据驱动接口回归, 自动提交问题、附件和账号, 收集 TraceID、响应和关键日志, 对权限、工具触发、缺参返回、RAG 是否触发做规则断言。开放回答部分再按关键事实点人工复核。

### Q57. Locust 如何支撑 AI Agent 性能测试?

参考答案: 按业务类型建不同任务权重: 普通问答、RAG 问答、工具任务、附件分析。分别统计入口耗时、首包、总耗时、错误率和下游耗时。AI 性能测试不能只给一个平均响应时间, 要拆到 RAG 和工具阶段。

### Q58. 你如何回答“你只是测试, 怎么理解 Agent 内部流程”?

参考答案: 测试不需要负责模型实现, 但必须理解产品链路和可观测节点。否则无法判断问题是权限、RAG、Prompt、工具、Observation 还是前端展示。我的角色是用流程图拆测试对象, 用日志和样本定位问题。

### Q59. 面试官问“你做过 RAG 吗”怎么回答?

参考答案: 我不会夸大说自己做过 RAG 算法开发。我会说项目里做过领域知识问答和知识库链路测试, 按新版流程理解就是 RAG 测试: 是否需要 RAG、Query Rewrite、权限过滤、TopK、ReRank、上下文注入、未命中处理和最终回答验证。

### Q60. 面试官问“这个项目你最大的测试价值是什么”怎么回答?

参考答案: 我的价值是把 AI Agent 从一个黑盒回答拆成可测试、可定位、可回归的链路。通过 TraceID、权限矩阵、RAG 样本、工具调用样本、异常返回样本和历史 badcase, 让问题能定位到具体节点, 让版本变更后能证明核心能力没有回退。
