# 1 天紧急准备版

## 使用目标

只准备命中率最高的问题。回答必须围绕新版流程图, 体现测试人员角色: 链路验证、权限验证、RAG 验证、工具调用验证、异常返回验证、TraceID 定位和回归样本沉淀。

## 必背项目主线

用户从 WeLink / Web / API 进入, UniKnow 做统一接入和账号绑定, 请求入口保留消息、附件和 TraceID。随后做身份、工具权限、知识库权限校验。通过后进入 Race Agent Service, 做统一编排、上下文和路由, 再通过意图识别判断普通问答、固定流程或复杂 Agent 任务。进入 Agent 后先判断目标、参数、附件、权限是否足够; 不足则返回缺失参数说明并结束本轮。上下文足够后判断是否需要 RAG, 需要时走 Query Rewrite、权限过滤、Retrieve/TopK、ReRank, 可用证据注入 Prompt/State。之后 Agent Executor 按 Plan / Thought / Action / Observation 执行, 进行 Tool Selection、参数组装与校验、API/MCP/工作流/子 Agent 调用, 根据 Observation 判断是否足够回答, 最后汇总工具结果和 RAG 依据生成最终回答。

## 问题与答案

### Q1. 请做一个自我介绍。追问: 为什么你适合元宝客户端测试工程师?

参考答案: 我有 7 年多测试和测试开发经验, 做过接口、性能、自动化、IoT 和 AI Agent 测试。最近的睿思 AI Agent 项目和元宝岗位比较贴近, 因为它既有客户端入口、附件上传、会话和异常返回, 也有 AI 链路里的鉴权、意图识别、RAG、Prompt/State、工具调用和结果生成。我在项目中是测试人员角色, 重点不是训练模型, 而是按链路验证每个节点是否可追踪、可定位、可回归。元宝客户端同样需要把“用户输入到最终回答”拆成客户端、接口、RAG、模型、工具和结果展示几层来保障质量。

### Q2. 你能按流程图讲一下睿思 AI Agent 的完整链路吗?

追问: 你实际负责哪些测试点?

参考答案: 链路分四段。第一段是接入鉴权分流: WeLink/Web/API 到 UniKnow, 请求入口保留消息、附件、TraceID, 再做身份、工具、知识库权限校验, 通过后由 Race Agent Service 做编排和意图识别。第二段是单轮前置判断和 RAG: 判断目标、参数、附件、权限是否足够, 不足就澄清并结束; 足够后判断是否需要 RAG, 需要时做 Query Rewrite、权限过滤、TopK 召回、ReRank。第三段是 Prompt/State 和 Agent 执行: 把输入、附件、RAG、权限构建成 State, 进入 Plan / Thought / Action / Observation。第四段是工具调用和结果返回。我负责验证这些节点的正确性、异常返回、日志证据和回归样本。

### Q3. TraceID 在这个流程里有什么用? 你怎么测?

追问: 如果线上用户反馈回答错误, TraceID 怎么帮助定位?

参考答案: TraceID 是跨接入层、Agent 服务、RAG、工具调用和最终回答的定位线索。我会验证请求入口是否生成或透传 TraceID, 后续 Race Agent Service、RAG 召回、Tool Call、Observation、最终回答日志里是否能串起来。线上反馈时, 我会用时间、账号、会话 ID 和 TraceID 查入口参数、权限结果、意图分流、RAG 命中、工具调用、Observation 和最终回答, 判断问题发生在哪一段。

### Q4. 权限校验怎么测?

追问: 身份、工具权限、知识库权限分别怎么覆盖?

参考答案: 身份权限测登录有效、登录失效、无权限用户、不同角色。工具权限测有权限用户能调用工具, 无权限用户不能触发工具或被明确拦截。知识库权限测不同部门、不同账号访问同一问题时, 是否只能召回自己有权限的知识。重点是权限失败必须在前面拦住, 不能让无权限知识进入 RAG, 也不能让无权限工具被 Agent 调用。

### Q5. 意图识别和任务分流怎么测?

追问: 普通问答、固定流程、复杂 Agent 任务如何区分?

参考答案: 我会准备三类样本。普通问答如业务概念解释, 不应进入 Agent 工具链路; 固定流程如标准操作指引, 可以走固定流程; 复杂任务如上传日志要求分析并给处理建议, 应进入 Agent 做规划和工具调用。验证时不只看回答, 还看分流日志、是否进入 Agent、是否触发 RAG 或工具。如果普通问答误进 Agent, 会增加耗时和误调用风险; 如果复杂任务没进 Agent, 可能只给泛化回答。

### Q6. 上下文是否足够这个节点怎么测?

追问: 缺参时为什么不能继续执行?

参考答案: 我会按目标、参数、附件、权限四类构造样本。比如用户说“帮我分析一下”但没有附件, 或上传日志但没有说明分析目标, 或工具需要业务域参数但输入里没有。期望是一次性返回缺失参数说明并结束本轮, 而不是继续猜测。因为单轮 Agent 的约束是不等待用户补充后恢复执行, 下一次用户输入会作为新请求重新进入流程。如果继续执行, 容易产生幻觉或错误工具调用。

### Q7. 单轮 Agent 缺参结束逻辑怎么验证?

追问: 这和多轮对话有什么区别?

参考答案: 我会验证缺参时是否只返回澄清问题, 不进入 RAG、Prompt/State 和工具调用。然后用户补充信息后, 要按新请求重新走接入、鉴权、上下文判断, 不能默认恢复上一次未完成任务。它和多轮对话不同: 多轮对话可以继承上下文, 但这个流程图强调单轮 Agent 缺参后结束, 所以测试时要看系统是否遵守这个产品逻辑。

### Q8. 是否需要 RAG 怎么判断? 你怎么测这个分支?

追问: 哪些问题不应该走 RAG?

参考答案: 需要企业知识、制度、规范、流程、接口文档、产品说明、内部字段、工具参数、业务概念映射的问题应该走 RAG。当前输入已经足够, 或主要依赖实时接口查询的问题可以不走 RAG。我会准备制度类、流程类、接口文档类、实时查询类样本, 看是否触发 RAG。误触发 RAG 可能引入无关知识, 漏触发 RAG 则可能回答无依据。

### Q9. Query Rewrite 怎么测?

追问: 改写后的 query 偏离用户原意怎么办?

参考答案: Query Rewrite 的目标是提升召回, 但不能改偏业务含义。我会准备缩写、同义词、口语化表达、带附件上下文的问题, 对比原始问题、改写结果和召回结果。比如用户问某个内部字段含义, 改写后应该保留字段名和业务域, 不能扩大成泛泛的流程问题。发现改写偏离时, 要记录原问题、改写 query、召回片段和最终回答, 作为 RAG badcase。

### Q10. RAG 权限过滤怎么测?

追问: 为什么权限过滤必须在召回前或召回过程中生效?

参考答案: 我会用不同部门、不同角色账号问同一个知识问题, 看无权限用户是否无法召回对应知识, 最终回答也不能泄露。权限过滤必须在知识进入 Prompt 前生效, 否则即使最终回答做了限制, 模型也可能已经看到无权限内容。测试时要看召回片段、RAG 上下文和最终回答三处证据。

### Q11. Retrieve / TopK 和 ReRank 怎么测?

追问: 召回命中但回答错, 你怎么判断问题在哪?

参考答案: TopK 看是否召回相关候选, ReRank 看最有用证据是否排到前面。我会准备标准问题、同义问法、干扰知识和冲突知识, 查看 TopK 结果、重排结果和最终注入片段。如果 TopK 没有正确片段, 是召回问题; TopK 有但 ReRank 没排上来, 是重排问题; 证据注入正确但回答错, 更可能是 Prompt/生成问题。

### Q12. RAG 未命中时应该怎么处理?

追问: 怎么防止模型编造答案?

参考答案: 未命中时应该不注入 RAG, 并标记知识不足或按普通执行处理。如果问题依赖企业知识, 最终回答应该说明无法确认或需要补充资料, 不能编造制度、流程或接口说明。我会用知识库不存在的问题、过期术语、无权限知识来测, 看系统是否把“未命中”传递到 Prompt/State 和最终回答。

### Q13. Prompt / State 构建怎么测?

追问: State 缺少 RAG 或权限信息会有什么后果?

参考答案: Prompt/State 应包含用户输入、附件解析结果、RAG 证据、权限状态和必要上下文。我会通过日志或调试信息确认这些内容是否进入 State。缺少 RAG 会导致回答无依据; 缺少权限会导致越权工具调用或越权回答; 缺少附件解析结果会导致日志分析变成泛化回答。

### Q14. Agent Executor 的 Plan / Thought / Action / Observation 怎么测?

追问: 测试人员怎么利用这些中间过程定位问题?

参考答案: 测试重点不是评价 Thought 写得好不好, 而是看执行链路是否合理。Plan 是否覆盖目标, Action 是否选择正确工具, Observation 是否记录工具结果或错误, 后续是否基于 Observation 决策。比如工具返回失败, Observation 应该记录失败状态, 后续不能生成成功结论。中间过程能帮助定位是规划错、工具错、参数错, 还是结果总结错。

### Q15. Tool Selection 怎么测?

追问: 如何判断误选工具或越权调用?

参考答案: 我会按权限和 Schema 准备样本。比如纯知识问答不应选日志分析工具; 上传日志分析应选对应工具; 无工具权限用户不能选工具。验证时看 Tool Selection 日志、权限校验结果和后续 Tool Call。如果选错工具, 可能导致错误业务结果; 如果越权调用, 属于高风险问题。

### Q16. 参数组装与校验怎么测?

追问: 参数来自用户输入、RAG、上下文时分别要注意什么?

参考答案: 我会校验参数来源、必填项、类型、枚举、业务域和权限。用户输入里的参数要防止抽取错; RAG 里的参数要防止引用无关知识; 上下文里的参数要防止串会话。比如工具需要设备类型、日志文件 ID、业务域, 如果缺一个就不应继续调用, 应返回缺失信息或校验失败原因。

### Q17. Tool Call 和 Observation 怎么测?

追问: 工具失败后最终回答应该怎样?

参考答案: Tool Call 要测 API/MCP/工作流/子 Agent 是否被正确调用, 入参是否正确, 超时和失败是否被记录。Observation 要记录工具结果、错误和业务状态。工具失败后最终回答不能说“已分析完成”, 应返回失败原因、缺失信息或建议重试。我会通过 TraceID 串 Tool Call 日志、Observation 和最终回答, 看三者是否一致。

### Q18. 结果是否足够回答这个判断怎么测?

追问: 什么情况下应该继续规划, 什么情况下应该终止?

参考答案: 如果工具结果和 RAG 证据已经覆盖用户目标, 就可以汇总回答; 如果缺关键证据、工具失败或参数不足, 应终止并返回原因; 如果本轮内部还有明确下一步, 可以继续规划。我会准备工具返回部分字段、空结果、错误码、超时、冲突结果的样本, 看系统是否错误地把不足结果当成充分结果。

### Q19. 最终回答怎么评估?

追问: 非确定性输出怎么断言?

参考答案: 最终回答要看答案、原因、下一步是否和工具结果、RAG 依据一致。开放式回答不能只做字符串完全匹配, 要按事实点、业务规则、证据引用和风险等级判断。比如日志分析类回答, 关键错误、故障方向、处理建议不能错; RAG 类回答, 制度、流程、接口字段必须有依据。

### Q20. 幻觉怎么测试?

追问: 回答很流畅但没有依据怎么办?

参考答案: 我会用无知识命中、缺附件、缺参数、工具失败、冲突证据等样本测试。期望是系统澄清、拒答或说明无法确认, 而不是编造结论。如果回答流畅但 RAG 未命中、工具未调用或 Observation 失败, 就是典型幻觉风险。我会把输入、TraceID、RAG 结果、Observation 和最终回答记录成 badcase。

### Q21. 线上 AI 质量问题怎么定位?

追问: 怎么拆客户端、接口、RAG、工具和模型问题?

参考答案: 先收集用户问题、截图、时间、账号、会话 ID、客户端版本、入口和 TraceID。再查客户端请求、附件上传、权限结果、意图分流、RAG 召回、Prompt/State、Tool Call、Observation、最终回答。如果客户端请求参数错, 是客户端或接入问题; RAG 证据错, 是检索问题; 工具结果对但回答错, 是生成或总结问题; Observation 失败但回答成功, 是结果判断问题。

### Q22. Prompt 变更后怎么回归?

追问: 新版流程下重点回归哪些节点?

参考答案: Prompt 变更后要重点回归缺参澄清、RAG 未命中、权限拒答、工具选择、参数校验、工具失败和最终回答格式。因为 Prompt 会影响是否编造、是否调用工具、是否正确利用 Observation。我会用历史缺陷、高频问答、权限边界、RAG 未命中、工具失败样本做分层回归, 不只看正常问题。

### Q23. 工具 Schema 变更后怎么回归?

追问: 参数名或枚举变了会影响什么?

参考答案: 工具 Schema 变更会影响 Tool Selection、参数组装、参数校验和 Tool Call。我会检查必填参数、类型、枚举、默认值、错误返回是否匹配。比如参数名变了但 Agent 仍按旧字段传, 工具可能失败; 枚举值变了, 可能调用成功但业务结果错。回归时要看 Tool Call 入参和 Observation。

### Q24. 元宝客户端和这个项目的相似点是什么?

追问: 你如何把这个项目经验迁移到元宝?

参考答案: 相似点是都不是单纯客户端页面测试, 核心是用户输入到 AI 回答的完整链路。元宝有客户端入口、会话、文件/图片、多模态、RAG、Prompt、工具或插件能力、线上反馈。我的项目经验可以迁移到链路拆解、TraceID 定位、RAG 证据验证、权限边界、工具调用、异常返回和回归样本沉淀。

### Q25. 如果面试官问你不是算法背景, 怎么回答?

追问: 你能做 AI 产品质量体系吗?

参考答案: 我会明确说我不是算法研发, 但我是 AI 应用测试和质量保障角色。我理解并测试过 Agent 链路里的意图分流、RAG、Prompt/State、工具调用、Observation、幻觉和线上 badcase。我的价值是把 AI 输出问题拆到接入、权限、RAG、Prompt、工具、接口或客户端展示, 用样本、日志和回归结果推动质量闭环。
