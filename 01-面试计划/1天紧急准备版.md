# 1 天紧急准备版

## 使用目标

只保留命中率最高的问题, 以真实项目经历口径组织回答。每个答案尽量体现睿思 AI Agent 场景、测试动作、定位证据、复测和沉淀。

## 问题与答案

### Q1. 请做一个自我介绍。追问: 为什么你适合元宝客户端测试工程师?

参考答案: 我有 7 年多测试和测试开发经验, 做过接口、性能、自动化、IoT 和 AI Agent 测试。最近的睿思 AI Agent 项目和元宝岗位比较贴近, 因为它不是普通功能测试, 而是要把用户输入到 AI 回答的链路拆开验证。比如用户从 WeLink 或 Browser 插件输入问题或上传日志, 后面会经过接入、鉴权、意图分流、知识库、工具调用、模型生成和结果回传。我在项目里主要做这些链路的测试、问题定位和回归样本沉淀。元宝客户端也类似, 既要测客户端交互、上传、会话、弱网和兼容性, 也要测 AI 回答准确性、RAG、Prompt、多轮上下文和线上质量闭环。

### Q2. 睿思 AI Agent 的完整业务链路是什么?

追问: 你实际负责哪几段?

参考答案: 链路是用户从 WeLink 或 Browser 插件进入, UniKnow 做账号、域名、附件和统一接入, Race Agent Service 做鉴权、上下文和意图识别, 然后按问题类型分流。简单规则类问题走 RaceRobot, 复杂任务走 Race Agent, 再调用知识库、API/MCP 插件、工作流、文件处理等能力, 最后汇总结果生成回答并回传。我负责的不是模型训练, 而是测试链路质量: 入口参数是否正确, 权限是否生效, 意图分流是否正确, 知识库是否命中, 工具是否正确触发, 工具结果是否被正确总结, 以及版本变更后的回归验证。

### Q3. 你在需求评审阶段做了什么?

追问: 如何识别 AI Agent 的质量风险?

参考答案: 需求评审时我会先看业务场景, 比如日志分析、分析件初筛、领域知识问答、多轮追问分别面向什么用户和什么结果。然后拆风险: 用户权限、附件上传、日志解析、知识库命中、工具调用、回答准确性、无法判断时是否拒答、历史缺陷是否会复现。AI Agent 的风险不像普通接口只看返回码, 还要看它是否编造结论、是否错误继承上下文、是否越权回答、是否把工具失败包装成成功结论。

### Q4. AI Agent 和普通问答机器人有什么区别?

追问: 测试重点有什么变化?

参考答案: 普通问答机器人更多是固定问答或知识匹配, AI Agent 会做任务规划、工具调用、多轮决策和外部系统交互。在睿思项目里, 一个日志分析问题不是直接生成回答, 它可能要识别意图、读取附件、调用分析工具、结合知识库再生成结论。所以测试重点要从“答案文本对不对”扩展到“分流路径对不对、工具有没有调、参数有没有传、工具结果有没有被正确使用、最终回答有没有依据”。

### Q5. 多轮对话怎么测?

追问: 上下文污染怎么发现?

参考答案: 我会设计连续对话场景。比如第一轮上传日志问“帮我分析这个问题”, 第二轮只问“那应该怎么处理”, 这时 Agent 应该继承上一轮日志分析结论。再设计话题切换, 比如第一轮问 A 故障, 第二轮切到 B 设备, 看它是否还套用 A 的结论。上下文污染主要通过干扰样本发现: 不同用户、不同群组、不同会话里输入相似问题, 看是否串上下文或引用无关历史信息。

### Q6. 工具调用怎么测?

追问: 如何判断误触发、漏触发、参数错误?

参考答案: 我会看三层: 触发条件、入参传递、执行结果。比如用户只是问业务概念, 不应该触发日志分析工具; 用户上传日志并要求分析, 就应该触发。触发后要看文件 ID、用户账号、业务域、会话 ID、问题类型是否传完整。如果该触发没触发是漏触发, 不该触发却触发是误触发, 触发了但日志里参数缺失或业务域错误就是参数组装问题。最终还要看工具返回失败时, Agent 有没有编造成功结论。

### Q7. Prompt 变更后怎么回归?

追问: 是否每次都全量回归?

参考答案: 我不会简单全量回归, 会先看 Prompt 改了哪里。系统 Prompt、工具描述、拒答规则这类变更影响大, 要覆盖核心问答、多轮上下文、工具调用、无效输入、权限边界和历史缺陷。局部措辞变更可以用重点样本集加抽样。睿思项目里, 我们会沉淀高频问答、异常日志、历史缺陷样本, 每次变更后对比关键事实、结论、工具调用路径和拒答边界有没有变化。

### Q8. 知识库变更后怎么测?

追问: 如何判断是召回问题还是生成问题?

参考答案: 我会先准备标准问题、同义问法、模糊问法、无答案问题和权限边界问题。测试时不仅看最终回答, 还会看命中的知识片段。如果没命中正确片段, 更像召回或切片问题; 如果命中了正确片段但回答错, 更像 Prompt 或模型生成问题; 如果命中了无权限内容, 就是权限隔离问题。睿思项目里的领域知识问答就很适合这样拆。

### Q9. 如何评估 AI 回答准确性?

追问: 非确定性输出怎么断言?

参考答案: 我不会要求输出逐字一致, 而是按事实点和业务规则判断。比如日志分析类回答, 我会看有没有识别关键错误信息、有没有给出正确故障方向、有没有遗漏必要前提、有没有编造日志里不存在的结论。业务问答类回答, 看流程步骤、适用条件和限制条件是否正确。非确定性输出可以允许表达不同, 但关键结论和事实不能错。

### Q10. 幻觉怎么测试?

追问: 如果回答看起来很流畅但事实错误怎么办?

参考答案: 幻觉就是模型生成了没有依据或事实错误的内容。在项目里我会用不存在的故障码、缺少关键字段的日志、知识库没有覆盖的问题、冲突知识样本来测。期望不是强行回答, 而是说明无法判断、要求补充信息或基于已有信息谨慎回答。如果回答很流畅但日志和知识库里没有依据, 我会记录输入、命中内容、工具结果和最终回答, 作为 badcase 推动修复。

### Q11. 超长日志、无效输入、模糊指令分别怎么测?

追问: 重点指标是什么?

参考答案: 超长日志重点看是否截断关键内容、是否还能提取错误信息; 无效输入看是否拒答或提示补充信息; 模糊指令看是否能澄清意图。比如用户只说“这个有问题吗”, 但没有上传日志, Agent 不应该编分析结论, 应该提示上传日志或补充现象。重点指标是准确性、一致性、鲁棒性、拒答合理性和用户可理解性。

### Q12. 线上 AI 质量问题怎么定位?

追问: 怎么拆前端、接口、RAG、模型、工具问题?

参考答案: 我会先拿用户问题、时间、账号、会话 ID、入口来源、附件、版本信息。然后复现输入, 查前端请求参数、Race Agent Service 日志、意图分流、知识库命中、工具调用日志、业务接口返回和最终回答。如果上传文件没有到预处理, 是接入或文件处理问题; 如果知识没命中, 是召回问题; 如果工具返回对但回答总结错, 是生成问题; 如果前端展示和接口返回不一致, 是客户端展示问题。

### Q13. 你的 Pytest 接口自动化框架怎么设计?

追问: fixture 和参数化怎么用?

参考答案: 我在平台和 IoT 服务端接口测试里用 Pytest 做过自动化。一般会分用例、测试数据、公共请求方法、环境配置、断言和报告。fixture 用来处理登录 token、公共前置数据、环境准备和清理; 参数化用来覆盖多组输入, 比如正常、异常、边界、权限不足。断言不只看 200, 还看业务码、关键字段、数据库状态和业务规则。

### Q14. 接口测试如何设计用例?

追问: 只断状态码够不够?

参考答案: 不够。比如数据分析平台接口返回 200, 但报表统计错或状态没流转, 仍然是缺陷。我会从正常流程、异常参数、边界值、权限、幂等、超时、重复提交、依赖服务失败设计用例。对核心接口, 还会结合数据库、日志和下游结果验证业务是否真的完成。

### Q15. Jenkins 自动化失败后怎么分析?

追问: 怎么区分环境问题和产品缺陷?

参考答案: 我会先看失败分布。如果大量用例同时失败, 优先怀疑环境、鉴权、网络、公共依赖或测试数据; 如果只有某个接口稳定失败, 就看请求参数、响应体、服务日志和最近代码变更。如果偶现失败, 再看超时、并发、外部依赖和数据污染。自动化不是只出报告, 关键是能帮助定位版本风险。

### Q16. Locust 性能测试怎么做?

追问: 关注哪些指标?

参考答案: 我会先确认压测目标, 比如并发用户数、业务比例、持续时长、目标响应时间和错误率。Locust 里按业务比例设计任务, 模拟用户等待时间, 逐步加压。关注平均响应、P90/P95/P99、错误率、RPS、吞吐和服务日志。AI Agent 场景还要额外看工具调用、知识库检索和模型生成耗时, 因为慢不一定慢在入口接口。

### Q17. P95 响应时间升高怎么排查?

追问: 数据库慢和下游慢怎么区分?

参考答案: 先看是全局慢还是单接口慢。全局慢看资源、网络、网关、线程池; 单接口慢看应用日志、慢 SQL、下游接口耗时、连接池和工具调用耗时。数据库慢通常能在慢 SQL、执行计划、扫描行数里看到; 下游慢会在外部调用耗时、超时日志里体现。测试侧要提供稳定复现请求和时间点。

### Q18. Linux 日志定位常用哪些命令?

追问: 如何按时间和 traceId 查链路?

参考答案: 常用 `grep`, `tail`, `less`, `awk`, `sed`, `top`, `df`, `du`, `ps`, `ss`。在项目里定位问题时, 我一般先按用户反馈时间和会话 ID 查入口日志, 再沿 traceId 查 Agent 服务、工具调用、业务接口日志。比如最终回答异常, 我会看请求进来时参数是什么, 中间分流到哪个服务, 工具有没有调用, 返回内容是什么。

### Q19. MySQL/PostgreSQL 在测试中怎么用?

追问: 如何验证数据准确性?

参考答案: 数据库主要用于验证数据落库、状态流转、权限数据、报表统计和异常定位。比如平台类需求里, 接口返回成功后, 我会查对应业务表状态、统计表结果、用户权限关系是否正确。AI Agent 场景如果涉及任务记录或调用记录, 也可以通过数据库辅助确认任务是否创建、状态是否更新、结果是否落库。

### Q20. 客户端出现“页面正常但回答异常”怎么定位?

追问: 元宝这种 AI 客户端要看哪些链路?

参考答案: 页面正常只能说明展示层没崩, 不代表 AI 链路正确。我会查客户端请求参数、会话 ID、用户身份、附件上传结果、接口返回、知识库命中、模型输出和前端渲染。元宝这种 AI 客户端还要看多端一致性、历史会话、文件/图片上传、弱网重试、回答流式输出、RAG、Prompt 和工具调用链路。

### Q21. RAG 是什么?

追问: RAG 的测试点有哪些?

参考答案: RAG 是先检索知识, 再基于知识生成回答。结合睿思项目, 领域知识问答就类似这个链路。测试点包括知识切片、召回、重排、引用、权限隔离、无答案拒答、冲突知识处理和更新回归。真实测试时, 我不会只看最终回答, 还会看它到底命中了哪些知识内容。

### Q22. Prompt Engineering 是什么?

追问: 测试人员需要懂到什么程度?

参考答案: Prompt 是通过系统指令、示例、约束和工具描述影响模型行为。测试人员不一定要像算法一样调模型, 但要知道 Prompt 会影响回答格式、拒答边界、工具选择、多轮上下文和幻觉风险。在项目里, Prompt 变更后我会用历史缺陷、高频问题、异常输入和工具调用样本做回归, 看有没有引入副作用。

### Q23. 多模态交互怎么测?

追问: 图片/文件上传异常怎么覆盖?

参考答案: 我之前主要做过文件/日志上传链路, 思路可以迁移到元宝多模态。要测上传成功、格式不支持、文件过大、空文件、损坏文件、解析失败、权限和弱网重试。比如日志文件上传成功但回答异常, 不能只看上传接口 200, 还要看文件预处理、解析结果、Agent 是否拿到正确内容。

### Q24. AI 产品全生命周期质量体系怎么搭?

追问: 左移和右移分别做什么?

参考答案: 左移是在需求、Prompt、知识库、接口协议、工具定义阶段提前评审风险; 测试中建立样本集、自动化回归和发布准入; 右移是上线后看用户反馈、badcase、点赞点踩、工具失败率、无答案率、响应耗时和线上日志。睿思项目里, 我做得比较多的是需求评审、回归样本沉淀、版本回归和线上问题定位。

### Q25. 你如何和产品、研发、算法协作?

追问: 算法认为不是 bug 怎么办?

参考答案: 我会用样本和证据沟通, 不只说“感觉不对”。比如给出用户输入、账号权限、知识库命中、工具调用日志、期望业务规则、实际回答和影响范围。如果算法认为是模型能力边界, 我会和产品确认用户预期和验收标准: 是接受边界提示, 还是需要 Prompt/知识库/工具链路优化。测试的价值是把问题复现清楚、边界说清楚。
